# Especificación de Requerimientos — SIBE

---

## 1. Requerimientos Funcionales

### 1.1 Gestión de Actividades y Áreas

| Código | Título | Descripción | Estado |
| ------ | ------ | ----------- | ------ |
| RF-001 | Gestión de Actividades por Dirección | El sistema deberá permitir al usuario con rol de Administrador de la Dirección crear y programar actividades. Para ello, se deberá disponer de una interfaz que posibilite asignar cada actividad a la Dirección, a un Área o a una Subárea de la Dirección de Bienestar y Evangelización, incluyendo la selección de fecha, hora e información adicional relevante. | Aprobado |
| RF-004 | Gestión de Actividades por Área | El sistema deberá permitir al usuario con rol de Administrador de Área crear y editar las actividades propias de su ámbito de acción mediante un panel de control específico, garantizando que solo se afecten las actividades correspondientes a su área o subárea asignada. | Aprobado |
| RF-006 | Navegación Integral entre Áreas | El sistema deberá facilitar la navegación entre las distintas áreas pertenecientes a la Dirección de Bienestar y Evangelización, permitiendo a los usuarios cambiar de contexto y visualizar la información específica de cada área mediante un menú o panel centralizado. | Aprobado |

### 1.2 Gestión de Usuarios y Acceso

| Código | Título | Descripción | Estado |
| ------ | ------ | ----------- | ------ |
| RF-002 | Creación de Usuarios | El sistema debe permitir al Administrador de Dirección crear nuevos usuarios con roles de Administrador de Dirección, Administrador de Área y Colaborador, mediante un formulario que capture los datos: nombres, apellidos, tipo de documento, número de documento, correo electrónico, contraseña y tipo de usuario. | Aprobado |
| RF-003 | Validación de Mínimo de Administradores | El sistema deberá asegurar que, al momento de la entrega, exista un mínimo de dos usuarios con perfil de Administrador de la Dirección, constituyendo esta condición previa para la aceptación del aplicativo. | Aprobado |
| RF-013 | Control de Acceso por Roles | El sistema deberá restringir el acceso a la información de cada área de la Dirección, permitiendo el ingreso únicamente a los usuarios asignados como administradores o colaboradores de esa área. Además, se deberá garantizar que el Administrador de la Dirección tenga acceso a la totalidad de las áreas para poder realizar funciones de supervisión y gestión global. | Aprobado |

### 1.3 Autenticación y Gestión de Credenciales

| Código | Título | Descripción | Estado |
| ------ | ------ | ----------- | ------ |
| RF-001-B | Inicio de Sesión | El sistema debe permitir a los usuarios iniciar sesión mediante un formulario que solicite el correo electrónico y la contraseña, validando las credenciales para otorgar acceso a la aplicación. | Aprobado |
| RF-003-B | Modificación de Usuarios | El sistema debe permitir al Administrador de Dirección modificar los datos de usuarios existentes a través de un formulario que permita actualizar nombres, apellidos, tipo de documento, número de documento, correo electrónico y tipo de usuario. | Aprobado |
| RF-004-B | Modificación de Contraseña | El sistema debe permitir a cada usuario modificar su propia contraseña mediante una opción segura dentro de la aplicación. | Aprobado |
| RF-005 | Eliminación/Deshabilitación de Usuarios | El sistema debe permitir al Administrador de Dirección eliminar usuarios que no estén relacionados con actividades registradas; en caso contrario, debe permitir deshabilitar al usuario para preservar el historial de gestión en la base de datos. | Aprobado |

### 1.4 Registro de Asistencia y Participantes

| Código | Título | Descripción | Estado |
| ------ | ------ | ----------- | ------ |
| RF-011 | Registro de Asistencia Mediante RFID | El sistema deberá integrar la funcionalidad de registro de asistencia a las actividades mediante un lector RFID que lea la información del carnet universitario, permitiendo así el control automático de la presencia en las actividades programadas. | Aprobado |
| RF-014 | Gestión de Participantes | El sistema deberá vincular los carnets universitarios leídos por el lector RFID a una base de datos que almacene la información de los integrantes activos (estudiantes, docentes y empleados). Dicha base de datos se actualizará de manera semestral para reflejar la vigencia de la información y asegurar la validez en el registro de asistencia. | Aprobado |

### 1.5 Visualización y Reportes

| Código | Título | Descripción | Estado |
| ------ | ------ | ----------- | ------ |
| RF-007 | Visualización de Gráficas Globales | El sistema deberá mostrar en la página principal gráficos globales que representen los datos de participación en las actividades realizadas por todas las áreas de la Dirección. Estos gráficos deberán estar actualizados de forma periódica y permitir al usuario obtener una visión consolidada del desempeño general. | Aprobado |
| RF-008 | Visualización de Gráficas Específicas por Área | El sistema deberá disponer de secciones en cada área que muestren gráficos detallados de participación en las actividades propias de dicha área, facilitando el análisis individualizado de resultados. | Aprobado |
| RF-009 | Filtrado de Datos en Gráficas | El sistema deberá permitir aplicar filtros a las gráficas generadas utilizando las siguientes variables: **Periodo:** año, semestre y mes. **Estudiantes:** diferenciación entre pregrado y posgrado, clasificados por facultad y programa, colegio MAUJ. **Empleados:** clasificación por tipo (docentes a tiempo completo, medio tiempo, ocasionales, cátedra; administrativos; operativos; directivos). **Externos:** asistentes sin credenciales de la universidad, como acudientes. Estos filtros facilitarán la personalización y segmentación del análisis de la información. | Aprobado |

## 2. Requerimientos No Funcionales

| Código | Atributo de Calidad | Título | Descripción | Estado |
| ------ | ------------------- | ------ | ----------- | ------ |
| RNF-001 | Usabilidad | Eficiencia en la Navegación | El sistema deberá garantizar que cualquier usuario pueda acceder a los módulos principales (creación de actividades, registro de asistencia, generación de gráficos) desde la página principal en máximo 3 clics, excluyendo pasos de autenticación inicial. | Aprobado |
| RNF-002 | Rendimiento | Tiempo de Respuesta en Consultas | El sistema deberá responder a las solicitudes de navegación entre áreas y carga de datos en menos de 2 segundos en condiciones de carga normal (hasta 200 usuarios concurrentes). | Aprobado |
| RNF-003 | Disponibilidad | Disponibilidad Operativa | La aplicación deberá estar disponible al menos el 99.9% del tiempo durante los periodos académicos activos (excluyendo mantenimiento programado previamente anunciado). | Aprobado |
| RNF-004 | Seguridad | Autenticación y Autorización | El sistema implementará autenticación mediante credenciales seguras (usuario/contraseña) con encriptación TLS 1.3+ y autorización basada en roles (Administrador de Dirección, Administrador de Área, Colaborador). Los permisos se gestionarán mediante políticas RBAC (Role-Based Access Control). | Aprobado |
| RNF-005 | Seguridad | Protección contra Ataques | El sistema deberá incluir mecanismos de protección contra vulnerabilidades comunes como SQL Injection, XSS y CSRF, validando todas las entradas de usuario y utilizando parámetros preparados en consultas a la base de datos. | Aprobado |
| RNF-006 | Mantenibilidad | Documentación Técnica | El sistema contará con documentación técnica detallada que incluya diagramas de arquitectura, flujos de datos y manuales de API para integraciones futuras. | Aprobado |
