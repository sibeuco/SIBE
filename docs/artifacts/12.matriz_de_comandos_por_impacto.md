# Matriz de Comandos por Impacto — SIBE

---

## 1. Descripción y Metodología

### 1.1 Propósito

Este artefacto documenta de forma exhaustiva cada **operación de escritura (comando)** del sistema SIBE, mapeando su flujo completo desde el punto de entrada HTTP hasta las mutaciones de persistencia, efectos secundarios y reglas de validación aplicadas. Permite responder a las preguntas:

- ¿Qué entidades se crean, modifican o eliminan cuando se ejecuta un comando?
- ¿Qué efectos secundarios dispara cada operación (email, encriptación, cambio de estado)?
- ¿Qué reglas de negocio se validan antes de persistir?
- ¿Cuáles son las cascadas JPA y la atomicidad transaccional de cada operación?

### 1.2 Metodología de Construcción

El análisis sigue la cadena completa de cada comando dentro de la arquitectura hexagonal con CQRS:

```
Controller (HTTP) → Manejador (Handler) → Fábrica (Factory) → UseCase (Dominio) → Servicios de Dominio → Puertos de Comando (Repository)
```

Se documentan las cascadas de persistencia JPA y las relaciones entre tablas.

### 1.3 Convenciones

| Símbolo | Significado |
| ------- | ----------- |
| **C** | CREATE — Se crea una nueva entidad/registro |
| **R** | READ — Se consulta la entidad (sin mutación) |
| **U** | UPDATE — Se actualiza un registro existente |
| **D** | DELETE — Se elimina físicamente un registro |
| ⚡ | Efecto secundario (email, encriptación, cambio de estado) |
| 🔒 | Requiere autorización contextual organizacional |
| ✅ | Validación de regla de negocio |

### 3.1 Gestión de Usuarios y Acceso

| Entidad | Guardar Usuario | Modificar Usuario | Eliminar Usuario | Modificar Clave | Solicitar Código | Validar Código | Recuperar Clave | Login |

**Detalle de operaciones compuestas:**

- **Guardar Usuario (C1):** Crea `Persona` + `Identificación` (cascade), `Usuario` + `UsuarioTipoUsuario` (cascade), y `UsuarioOrganización` (vinculación a Dirección/Área/Subárea según `TipoArea`). Encripta la contraseña vía `EncriptarClaveServicio`.
- **Modificar Usuario (C2):** Actualiza `Persona` + `Identificación`, actualiza `Usuario` + `UsuarioTipoUsuario`, y modifica vinculación organizacional vía `ModificarVinculacionUsuarioConAreaService` (desvincula la anterior, vincula la nueva).
- **Eliminar Usuario (C3):** Realiza un borrado lógico (desactivación) del `Usuario`, marcándolo como inactivo en lugar de eliminarlo físicamente. Si se vuelve a crear un usuario con el mismo documento de identidad, se reactiva el registro existente. Valida acceso contextual 🔒.

**Detalle de operaciones compuestas:**

- **Guardar Actividad (C9):** Crea `Actividad`, luego crea N `EjecuciónActividad` (una por cada fecha programada), cada una con estado `PENDIENTE` (cascade a `EjecuciónActividadEstadoActividad`). Vincula la actividad al Área/Subárea/Dirección correspondiente vía `VincularActividadConAreaService`.
- **Modificar Actividad (C10):** Actualiza `Actividad`, actualiza `EjecuciónActividad` existentes (por identificador) y crea nuevas ejecuciones (sin identificador). Desvincula de Área anterior y vincula a la nueva vía `ModificarVinculacionActividadConAreaService`.
- **Finalizar Actividad (C12):** Cambia estado de `EjecuciónActividad` a `FINALIZADA`, establece `fechaRealizacion` y `horaFin`. **OPERACIÓN BULK:** Itera sobre la lista de participantes, registra cada uno vía `RegistrarParticipanteService` (upsert por documento) y crea `RegistroAsistencia` por cada participante.

**Detalle de operaciones compuestas:**

- **Cargar Masivamente Estudiantes (C20):** Procesa archivo Excel vía `ProcesadorArchivoEstudianteServicio`, mapea cada fila a `DatosEstudianteComando`, construye `Estudiante` mediante `EstudianteFabrica`. El UseCase hace **upsert**: busca por número de identificación; si existe, modifica; si no, crea. Durante el upsert se actualiza el campo `estaActivo` para reflejar el estado vigente del estudiante en la carga. Se valida cada registro con `MotoresFabrica.MOTOR_ESTUDIANTE` y `MOTOR_CIUDAD_RESIDENCIA`. Solo el **Administrador de Dirección** puede ejecutar esta operación.
- **Cargar Masivamente Empleados (C21):** Mismo patrón que estudiantes. El upsert también gestiona el campo `estaActivo` del empleado. Valida con `MOTOR_EMPLEADO`, `MOTOR_CIUDAD_RESIDENCIA`, `MOTOR_RELACION_LABORAL` y `MOTOR_CENTRO_COSTOS`. Las entidades `EmpleadoRelacionLaboral` y `EmpleadoCentroCostos` tienen cascade ALL desde `EmpleadoEntidad`. Solo el **Administrador de Dirección** puede ejecutar esta operación.

## 4. Transiciones de Estado

### 4.1 Máquina de Estados: EjecuciónActividad

```
┌─────────────┐    IniciarActividad    ┌─────────────┐    FinalizarActividad    ┌──────────────┐
│  PENDIENTE   │ ────────────────────► │  EN_CURSO   │ ──────────────────────► │ FINALIZADA   │
│  (Inicial)   │                       │             │                         │  (Terminal)   │
└─────────────┘                       └──────┬──────┘                         └──────────────┘
                                              │
                                              │ CancelarActividad
                                              │ (Rollback)
                                              ▼
                                       ┌─────────────┐
                                       │  PENDIENTE   │
                                       │  (Revertido) │
                                       └─────────────┘
```

| Transición | Comando | Estado Origen | Estado Destino | Campos Mutados |
| ---------- | ------- | ------------- | -------------- | -------------- |
| Iniciar | `IniciarActividadUseCase` | PENDIENTE | EN_CURSO | `estado`, `horaInicio` ← `LocalTime.now()` |
| Finalizar | `FinalizarActividadUseCase` | EN_CURSO | FINALIZADA | `estado`, `fechaRealizacion` ← `LocalDate.now()`, `horaFin` ← `LocalTime.now()` |
| Cancelar | `CancelarActividadUseCase` | EN_CURSO | PENDIENTE | `estado` |

| ------- | ---------------- | ------------------ |

| Finalizar Actividad | EN_CURSO | `ValorInvalidoExcepcion` — "estado no es EN_CURSO" |
| Cancelar Actividad | EN_CURSO | `ValorInvalidoExcepcion` — "estado no es EN_CURSO" |

### 5.2 Clasificación de Riesgo de Efectos Secundarios

| Nivel de Riesgo | Efecto | Justificación |
| --------------- | ------ | ------------- |
| 🔴 **Alto** | Envío de correo electrónico (C5) | Comunicación externa irreversible. Fallo en SMTP puede dejar estado inconsistente (código creado sin email enviado). Mitigación: `@Async` desacopla la transacción |
| 🔴 **Alto** | Registro masivo de participantes (C12) | N inserciones en una transacción. Fallo a mitad puede dejar participantes parcialmente registrados |
| 🟡 **Medio** | Procesamiento de Excel (C20, C21) | Archivo externo potencialmente grande. Validación fila por fila. Error en una fila afecta toda la carga |
| 🟢 **Bajo** | Encriptación de clave | Operación local, determinista, sin efecto externo |
| 🟢 **Bajo** | Captura de timestamp | Operación local usando reloj del servidor |

### 8.2 Lógica de Autorización Contextual (`AutorizacionContextoOrganizacionalServicio`)

La autorización contextual sigue una jerarquía de permisos:

1. **ADMINISTRADOR_DIRECCION**: Acceso total a todos los recursos (bypass completo)
2. **ADMINISTRADOR_AREA**: Puede estar asignado a un **Área** o a una **Subárea** específica:
   - Si está asignado a un **Área**: accede a los recursos de su área y todas las subáreas que pertenecen a esa área.
   - Si está asignado a una **Subárea**: accede únicamente a los recursos de esa subárea específica.
3. **COLABORADOR**: Puede estar asignado a un **Área** o a una **Subárea** específica:
   - Si está asignado a un **Área**: accede a los recursos de su área y las subáreas de esa área.
   - Si está asignado a una **Subárea**: accede únicamente a los recursos de esa subárea.

| Método de Validación | Lógica |
| -------------------- | ------ |
| `validarAccesoADireccion(id)` | Solo ADMIN_DIRECCION pasa |
| `validarAccesoAArea(areaId)` | ADMIN_DIRECCION pasa siempre. ADMIN_AREA pasa si `areaId == contexto.areaId` (cuando está asignado a un área). COLABORADOR pasa si `areaId == contexto.areaId` (cuando está asignado a un área) |
| `validarAccesoASubarea(subareaId)` | ADMIN_DIRECCION pasa. ADMIN_AREA pasa si la subárea pertenece a su área (cuando está asignado a un área) o si `subareaId == contexto.subareaId` (cuando está asignado directamente a una subárea). COLABORADOR pasa si la subárea pertenece a su área asignada, o si `subareaId == contexto.subareaId` (cuando está asignado directamente a una subárea) |
| `validarAccesoAActividad(actividadId)` | ADMIN_DIRECCION pasa. ADMIN_AREA pasa si la actividad está en su área o en una subárea de su área (asignado a área), o si la actividad está en su subárea (asignado a subárea). COLABORADOR pasa si la actividad está en su área/subáreas (asignado a área), o en su subárea (asignado a subárea) |
| `validarAccesoAEjecucionActividad(id)` | Delega a `validarAccesoAActividad(ejecucion.actividad.id)` |
| `validarAccesoAUsuario(usuarioId)` | ADMIN_DIRECCION pasa. ADMIN_AREA pasa si el usuario pertenece a su área (asignado a área) o a su subárea (asignado a subárea). COLABORADOR pasa si es él mismo |
