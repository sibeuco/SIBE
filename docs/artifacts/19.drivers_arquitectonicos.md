## 1. Descripción del Artefacto

Este artefacto documenta los **drivers arquitectónicos** del sistema SIBE (Sistema de Información de Bienestar y Evangelización): las fuerzas que moldean y justifican las decisiones de diseño del sistema. Un driver arquitectónico es cualquier factor — sea un objetivo de negocio, un requisito de calidad, una restricción tecnológica o un riesgo — que influye de manera significativa en la estructura del software.

El documento se organiza según el modelo estándar de la industria:

1. **Drivers de Negocio**: objetivos y necesidades de los stakeholders que dan dirección al sistema.
2. **Atributos de Calidad**: requisitos no funcionales expresados como escenarios medibles (modelo SEI/Bass).
3. **Restricciones Técnicas**: limitaciones no negociables impuestas por la organización, la tecnología o el entorno.
4. **Requerimientos Funcionales Arquitectónicamente Significativos**: funcionalidades cuya complejidad o impacto obliga a decisiones estructurales.
5. **Decisiones Arquitectónicas**: las respuestas de diseño trazadas a los drivers que las motivaron.
6. **Riesgos y Sensibilidades**: puntos donde un cambio en los supuestos podría invalidar decisiones actuales.
7. **Matriz de Trazabilidad**: mapeo bidireccional driver → decisión → implementación.

## 3. Atributos de Calidad

Los atributos de calidad se expresan como **escenarios** siguiendo el modelo de Bass/SEI (fuente, estímulo, artefacto, entorno, respuesta, medida de respuesta).

### AC-01. Usabilidad — Navegación Eficiente

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Usuario final (cualquier rol) |
| **Estímulo** | Acceder a cualquier módulo principal del sistema |
| **Artefacto** | Interfaz web del frontend Angular |
| **Entorno** | Operación normal |
| **Respuesta** | El usuario alcanza el módulo destino |
| **Medida** | Máximo 3 clics desde la página principal |
| **Fuente documental** | Artefacto 15 (Requerimientos) |

### AC-02. Rendimiento — Tiempo de Respuesta

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Usuario autenticado |
| **Estímulo** | Navegación entre áreas y carga de datos |
| **Artefacto** | Backend API + Frontend SPA |
| **Entorno** | Carga normal (hasta 200 usuarios concurrentes) |
| **Respuesta** | Los datos se presentan en pantalla |
| **Medida** | Menos de 2 segundos |
| **Fuente documental** | Artefacto 15 |

### AC-03. Disponibilidad — Periodos Académicos

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Comunidad universitaria |
| **Estímulo** | Acceso al sistema durante periodo académico |
| **Artefacto** | Sistema SIBE completo (backend + frontend + BD) |
| **Entorno** | Periodos académicos activos |
| **Respuesta** | El sistema está operativo y accesible |
| **Medida** | 99.9% de disponibilidad, excluyendo mantenimientos programados |
| **Fuente documental** | Artefacto 15 |

### AC-04. Seguridad — Autenticación

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Usuario no autenticado |
| **Estímulo** | Intento de acceso al sistema |
| **Artefacto** | Cadena de filtros de Spring Security |
| **Entorno** | Cualquier momento |
| **Respuesta** | Autenticación via HTTP Basic + generación de JWT firmado con HMAC-SHA256 |
| **Medida** | Solo credenciales válidas de usuarios activos obtienen acceso; cifrado de claves con BCrypt; transmisión sobre TLS 1.3+ |
| **Fuente documental** | Artefactos 4, 15, 17 |

### AC-05. Seguridad — Autorización de Doble Capa

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Usuario autenticado |
| **Estímulo** | Operación sobre un recurso del sistema |
| **Artefacto** | Controladores + `AutorizacionContextoOrganizacionalServicio` |
| **Entorno** | Cualquier operación de comando o consulta |
| **Respuesta** | Se valida primero el rol (`@PreAuthorize`) y luego el acceso organizacional (Dirección/Área/Subárea) |
| **Medida** | Ningún rol accede a datos fuera de su ámbito organizacional; 0 bypasses de autorización contextual |
| **Fuente documental** | Artefactos 11, 12, 15, 17 |

### AC-06. Seguridad — Protección contra Ataques

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Actor malicioso externo o interno |
| **Estímulo** | Intento de SQL Injection, XSS o CSRF |
| **Artefacto** | Backend API |
| **Entorno** | Producción |
| **Respuesta** | Validación de entradas via motores de dominio, consultas parametrizadas JPA, API stateless sin CSRF |
| **Medida** | 0 vulnerabilidades OWASP Top 10 explotables |
| **Fuente documental** | Artefacto 15 |

### AC-07. Mantenibilidad — Arquitectura Hexagonal

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Equipo de desarrollo |
| **Estímulo** | Cambio en lógica de negocio, infraestructura o framework |
| **Artefacto** | Dominio aislado de Spring e infraestructura |
| **Entorno** | Desarrollo / evolución |
| **Respuesta** | Cambios en infraestructura no requieren modificar el dominio y viceversa |
| **Medida** | 0 imports de Spring/JPA en paquetes de dominio; use cases registrados como beans via `@Configuration` explícita |
| **Fuente documental** | Coding Standards, Artefacto 2 |

### AC-08. Mantenibilidad — Separación CQRS

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Equipo de desarrollo |
| **Estímulo** | Modificar lógica de escritura sin afectar consultas o viceversa |
| **Artefacto** | Controladores, manejadores y casos de uso separados por comando/consulta |
| **Entorno** | Desarrollo / evolución |
| **Respuesta** | Los paths de comando y consulta son independientes en todas las capas |
| **Medida** | Controladores, manejadores, puertos y casos de uso en paquetes separados por tipo de operación |

### AC-09. Mantenibilidad — Modularidad Frontend

| Aspecto | Descripción |

| **Entorno** | Desarrollo / evolución |
| **Respuesta** | Cambios confinados al módulo Feature afectado sin impactar Core ni Shared |

| Aspecto | Descripción |

| **Entorno** | Build pipeline |
| **Respuesta** | Tests unitarios por capa arquitectónica con reportes de cobertura |
| **Medida** | Coverage HTML/XML/CSV generado; tests alineados con estructura hexagonal |
| **Fuente documental** | Artefacto 14, Coding Standards |

### AC-11. Integridad de Datos — Registros Históricos Inmutables

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Proceso de finalización de actividad |
| **Estímulo** | Registro de asistencia de participantes |
| **Artefacto** | `RegistroAsistencia`, `Participante` |
| **Entorno** | Operación normal |
| **Respuesta** | Snapshot inmutable del participante asociado al registro de asistencia |

| **Fuente documental** | Artefactos 5, 7, 15 |

### AC-12. Confiabilidad — Atomicidad Transaccional

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Operación de comando (escritura) |
| **Estímulo** | Fallo parcial durante una transacción multi-paso |
| **Artefacto** | Interfaces `ManejadorComandoRespuesta`, `ManejadorParametroRespuesta`, `ManejadorComandoParametroRespuesta` |
| **Entorno** | Operación normal |
| **Respuesta** | `@Transactional` en la interfaz del manejador garantiza rollback automático ante cualquier excepción |
| **Medida** | Ningún dato parcial persiste en caso de fallo; cargas masivas aplican transacción atómica completa |
| **Fuente documental** | Artefactos 14, 17 |

| Aspecto | Descripción |
|---------|-------------|

| **Estímulo** | Carga semestral de archivo Excel con estudiantes o empleados |

| **Entorno** | Inicio de semestre (operación periódica) |
| **Respuesta** | Procesamiento completo del archivo con lógica upsert por identificación |
| **Medida** | Procesamiento exitoso sin degradación excesiva ni timeout |
| **Fuente documental** | Artefactos 4, 14 |

### AC-14. Confidencialidad — Aislamiento Organizacional

| Aspecto | Descripción |
|---------|-------------|
| **Fuente** | Usuario con rol limitado (Administrador de Área, Colaborador) |
| **Estímulo** | Intento de acceder a datos de otra unidad organizacional |
| **Artefacto** | `AutorizacionContextoOrganizacionalServicio` + claims del JWT |
| **Entorno** | Cualquier operación |
| **Respuesta** | Denegación de acceso con HTTP 403 |
| **Medida** | 0 fugas de datos entre unidades organizacionales |
| **Fuente documental** | Artefactos 4, 11, 12 |

## 4. Restricciones Técnicas

| ID | Restricción | Tipo | Justificación | Implementación Verificada |

| RT-04 | **Angular 16.2.x** como framework frontend | Stack | SPA con modularidad, lazy loading y ecosistema de testing | `package.json`: `"@angular/core": "^16.2.0"` |
| RT-05 | **PostgreSQL** como base de datos relacional de producción | Stack | BD robusta, gratuita y compatible con JPA/Hibernate | `application.properties`: `jdbc:postgresql://localhost:5432/sibe_db2` |
| RT-06 | **H2** como base de datos para entorno de desarrollo/testing | Stack | Testing rápido sin infraestructura externa | `build.gradle`: `runtimeOnly 'com.h2database:h2'` |
| RT-07 | **Despliegue en servidores internos de la universidad** | Infraestructura | Política institucional de la universidad | CORS configurado para `localhost:4200`; requiere reconfiguración para producción |

| RT-11 | **Arquitectura Hexagonal** como estilo arquitectónico backend | Estructural | Aislamiento del dominio de frameworks y detalles de infraestructura | Paquetes `dominio`, `aplicacion`, `infraestructura` con puertos e implementaciones separados |
| RT-12 | **CQRS** como patrón de separación comando/consulta | Estructural | Independencia de paths de lectura y escritura en todas las capas | Controladores, manejadores y puertos separados por tipo |
| RT-13 | **Use cases registrados via `@Configuration`**, no por escaneo de componentes | Estructural | Inyección explícita de puertos de dominio; dominio libre de anotaciones Spring | `UseCaseBean.java` con `@Bean` por cada caso de uso |
| RT-14 | **Apache POI 5.4.0** para procesamiento de archivos Excel | Stack | Motor de lectura de workbooks para carga masiva | `build.gradle`: `implementation 'org.apache.poi:poi-ooxml:5.4.0'` |
| RT-15 | **Spring Mail** para envío de correo institucional (recuperación de clave) | Stack | Integración SMTP para flujo de OTP | `application.properties`: SMTP Gmail puerto 587 |

## 6. Decisiones Arquitectónicas Clave

Cada decisión se documenta con su contexto, alternativas consideradas, justificación y trazabilidad a los drivers que la motivaron.

### DA-01. Arquitectura Hexagonal con Puertos y Adaptadores

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | El dominio de negocio es complejo (jerarquía organizacional, máquinas de estado, múltiples agregados) y debe ser testeable independientemente del framework |
| **Decisión** | Organizar el backend en tres capas: `dominio` (modelos, puertos, use cases, reglas), `aplicacion` (manejadores, fábricas, comandos), `infraestructura` (controladores, DAOs, seguridad, configuración) |
| **Alternativas descartadas** | Arquitectura en capas tradicional (acoplamiento dominio-infraestructura); Clean Architecture pura (sobrecarga de capas para el tamaño del proyecto) |
| **Justificación** | Aislamiento real del dominio (0 imports de Spring en `dominio`); facilita testing unitario de use cases con mocks de puertos; permite cambiar BD o framework sin tocar lógica de negocio |
| **Drivers** | AC-07, RT-11, DN-10 |
| **Implementación** | Paquetes `co.edu.uco.sibe.dominio`, `.aplicacion`, `.infraestructura`; puertos en `dominio.puerto.comando` y `dominio.puerto.consulta`; adaptadores en `infraestructura.adaptador.repositorio` |

### DA-02. CQRS Estructural en Todas las Capas

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Las operaciones de escritura tienen lógica compleja (validaciones, autorizaciones, side effects) mientras que las lecturas son simples consultas filtradas |
| **Decisión** | Separar controladores, manejadores, puertos y casos de uso en paquetes de `comando` y `consulta` |
| **Alternativas descartadas** | Controlador único por recurso (mezcla responsabilidades); CQRS con event sourcing (sobrecarga para el alcance del proyecto) |
| **Justificación** | Independencia de evolución entre paths de lectura y escritura; transaccionalidad solo en manejadores de comando; claridad en la asignación de permisos |
| **Drivers** | AC-08, RT-12 |
| **Implementación** | `infraestructura.controlador.comando.*` vs `infraestructura.controlador.consulta.*`; `aplicacion.comando.manejador.*` vs `aplicacion.consulta.*`; interfaces genéricas `ManejadorComandoRespuesta<C,R>`, `ManejadorParametroRespuesta<P,R>`, etc. |

### DA-03. Transaccionalidad Declarativa en Interfaces de Manejador

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Todas las operaciones de escritura deben ser atómicas, incluyendo las que involucran múltiples agregados (ej: carga masiva, finalización con participantes) |
| **Decisión** | Declarar `@Transactional` en la interfaz del manejador de comando, no en el caso de uso ni en el repositorio |
| **Alternativas descartadas** | `@Transactional` en use cases (introducir dependencia Spring en dominio); `@Transactional` en repositorios (transacciones demasiado granulares) |
| **Justificación** | El manejador es el punto de orquestación entre fábrica y caso de uso; la transacción abarca toda la operación de negocio; el dominio permanece libre de anotaciones Spring |
| **Drivers** | AC-12, AC-07, RT-13 |
| **Implementación** | `ManejadorComandoRespuesta<C,R>` con `@Transactional` en el método `ejecutar()`; consultas con `@Transactional(readOnly = true)` en repositorios |

### DA-04. Motor de Validación por Operación (MotoresFabrica)

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Las reglas de validación varían según el tipo de operación (crear vs. actualizar) y deben aplicarse consistentemente en todos los casos de uso |
| **Decisión** | Centralizar reglas de validación en motores pre-construidos estáticamente en `MotoresFabrica`, ejecutados por tipo de operación (`TipoOperacion.CREAR`, `TipoOperacion.ACTUALIZAR`) |
| **Alternativas descartadas** | Validación en controladores (Spring Validation — mezcla validación de negocio con infraestructura); validación ad-hoc en cada use case (duplicación e inconsistencia) |
| **Justificación** | Reglas de dominio centralizadas y reutilizables; validación diferenciada por contexto de operación; dominio autodescriptivo sin dependencias externas |
| **Drivers** | AC-07, AC-06 |
| **Implementación** | `MotoresFabrica` con `MOTOR_USUARIO`, `MOTOR_PERSONA`, `MOTOR_IDENTIFICACION`, `MOTOR_ACTIVIDAD`, `MOTOR_EJECUCION_ACTIVIDAD`, `MOTOR_ACCION`, `MOTOR_PROYECTO`, `MOTOR_INDICADOR`, `MOTOR_ESTUDIANTE`, `MOTOR_EMPLEADO`, `MOTOR_CIUDAD_RESIDENCIA`, `MOTOR_RELACION_LABORAL`, `MOTOR_CENTRO_COSTOS`; cada motor construido por su `*MotorFabrica` con reglas agregadas por `TipoOperacion` |

### DA-05. Generación de UUID con Anti-Colisión

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Todos los identificadores de entidades son UUID generados por la aplicación (no por la BD) |
| **Decisión** | La generación de UUID incluye un loop de verificación contra el repositorio para evitar colisiones |
| **Alternativas descartadas** | UUID secuencial (expone información de orden); auto-increment de BD (acoplamiento con motor de BD); UUID sin verificación (riesgo teórico de colisión) |
| **Justificación** | UUIDs aleatorios sin exponer orden ni volumen de datos; verificación explícita elimina riesgo de colisión; generación en dominio mantiene independencia de BD |
| **Drivers** | AC-07, AC-11 |
| **Implementación** | `UtilUUID.generar(Predicate<UUID> verificadorExistencia)` — loop `do-while` que genera UUIDs hasta encontrar uno no existente |

### DA-06. Seguridad Stateless con JWT + HTTP Basic

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | API REST consumida por SPA Angular; no se requiere estado de sesión en servidor |
| **Decisión** | Autenticación initial via HTTP Basic; generación de JWT firmado con HMAC-SHA256 conteniendo claims de identidad, rol y contexto organizacional; validación del JWT en cada request posterior |
| **Alternativas descartadas** | Sesiones server-side (no escala, no es REST); OAuth2/OIDC (sobrecarga para autenticación interna sin IdP externo) |
| **Justificación** | Stateless permite horizontal scaling; JWT porta claims evitando queries redundantes; HMAC-SHA256 es eficiente para firma simétrica en entorno de servidor único |
| **Drivers** | AC-04, RT-09, RT-10 |
| **Implementación** | `ProjectSecurityConfig` con `SessionCreationPolicy.STATELESS`; cadena: `ExceptionFilter` → `RequestValidationBeforeFilter` → `JWTTokenValidatorFilter` → `BasicAuth` → `JWTTokenGeneratorFilter`; claims: email, id, authorities, rol, direccionId, areaId, subareaId; expiración: 30,000,000 ms (~8.33h) |

### DA-07. Autorización en Dos Capas (Rol + Contexto Organizacional)

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | No basta con verificar el rol del usuario; también se debe asegurar que opera dentro de su unidad organizacional asignada |
| **Decisión** | Implementar dos capas de autorización: (1) `@PreAuthorize` con expresiones SpEL para verificar rol/permisos; (2) `AutorizacionContextoOrganizacionalServicio` en el caso de uso para verificar que el recurso pertenece al ámbito del usuario |
| **Alternativas descartadas** | Single-layer RBAC (no cubre segregación organizacional); ACL de Spring Security (sobrecarga de configuración); row-level security en BD (acoplamiento con motor de BD) |
| **Justificación** | La doble capa es inherente al modelo de negocio: un Administrador de Área tiene permisos de escritura pero solo sobre recursos de su área; la verificación organizacional es lógica de dominio, no infraestructura |
| **Drivers** | AC-05, AC-14, RF-AS-02 |
| **Implementación** | Capa 1: constantes `@PreAuthorize` en controladores (ej: `HAS_AREA_ADMIN_OR_ADMIN_CREATE_AUTHORITY`); Capa 2: `AutorizacionContextoOrganizacionalServicio.validarAccesoAArea()`, `.validarAccesoAUsuario()`, `.validarAccesoAEjecucionActividad()` — extrae claims del JWT via `SecurityContextHolder` y compara con la jerarquía organizacional |

### DA-08. Soft Delete para Usuarios

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Los usuarios pueden tener registros históricos de actividades y asistencias que deben preservarse para auditoría |
| **Decisión** | Implementar borrado lógico con campo booleano `estaActivo` en lugar de DELETE físico; filtrar usuarios inactivos en todas las consultas de repositorio |
| **Alternativas descartadas** | Hard delete con cascade (pérdida de datos históricos); tabla de archivo separada (complejidad de migración); flags por tabla de auditoría (inconsistencia potencial) |
| **Justificación** | Preserva integridad referencial de registros de asistencia; mantiene trazabilidad completa; patrones de filtrado centralizados en la implementación del repositorio |
| **Drivers** | AC-11, RF-AS-06 |
| **Implementación** | `UsuarioEntidad.estaActivo` (boolean); `PersonaRepositorioConsultaImplementacion` verifica `isEstaActivo()` y retorna `null` si es `false`; efecto cascada: usuario inactivo no puede autenticarse, no aparece en consultas, no puede recuperar clave |

### DA-09. Datos de Referencia Sembrados por DataLoaders

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | El sistema requiere datos de catálogo pre-existentes (tipos de usuario, estados de actividad, público de interés, etc.) y un usuario administrador inicial |
| **Decisión** | Usar `CommandLineRunner` con `@Order` para sembrar datos de referencia al inicio de la aplicación; Hibernate `ddl-auto=update` para evolución del schema |
| **Alternativas descartadas** | Flyway/Liquibase (no implementado aún; mayor disciplina de versionado); scripts SQL manuales (propenso a errores); data.sql de Spring Boot (menos flexibilidad que code) |
| **Justificación** | DataLoaders ejecutan lógica Java (ej: BCrypt del password del admin seed); `@Order` garantiza secuencia correcta; idempotentes por diseño (verifican existencia antes de insertar) |
| **Drivers** | RT-05, DN-10 |
| **Implementación** | `DataLoader` abstracto implementa `CommandLineRunner`; `TipoUsuarioDataLoader` (`@Order(1)`), `EstadoActividadDataLoader`, `UsuarioAdministradorDataLoader` (`@Order(10)`); schema evoluciona via `hibernate.ddl-auto=update` |

### DA-10. Arquitectura Frontend Core/Shared/Feature con Lazy Loading

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | El frontend debe ser modular, con dominios funcionales independientes y tiempo de carga inicial mínimo |
| **Decisión** | Organizar en tres capas: `CoreModule` (interceptores, guards, error handler — singleton), `SharedModule` (componentes reutilizables, StateService), `Feature modules` (lazy-loaded por ruta) |
| **Alternativas descartadas** | Monolito sin módulos (no escala); micro-frontends (sobrecarga para equipo pequeño); NgRx (demasiada ceremonia para el alcance) |
| **Justificación** | Lazy loading reduce bundle inicial; separación Core/Shared/Feature evita dependencias circulares; cada feature module es autónomo |
| **Drivers** | AC-09, AC-01, RT-04 |
| **Implementación** | `AppModule` imports `CoreModule` + `SharedModule`; routing: `loadChildren: () => import('./feature/home/home.module')` con `canActivate: [securityGuard]`; routing anidado para área/subárea modules |

### DA-11. Estado Reactivo en Frontend con BehaviorSubject

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | El frontend necesita compartir estado de sesión y datos de usuario entre componentes sin acoplamiento directo |
| **Decisión** | `StateService` centralizado basado en `BehaviorSubject` de RxJS con rehidratación desde `sessionStorage` |
| **Alternativas descartadas** | NgRx Store (sobrecarga); localStorage directo (no reactivo); Signals de Angular 16 (no adoptados en el proyecto) |
| **Justificación** | BehaviorSubject es ligero, reactivo y basta para el tamaño del estado del sistema; rehidratación permite sobrevivir recargas de página; sessionStorage se destruye al cerrar pestaña (seguridad implícita) |
| **Drivers** | AC-04, DN-06 |
| **Implementación** | `StateService` en `SharedModule`; `sessionStorage['Authorization']` como almacenamiento persistente del JWT; guards leen del StateService para decisiones de routing |

### DA-12. Doble Interceptor para Autenticación Frontend

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | El frontend debe enviar HTTP Basic para login y JWT Bearer para requests subsiguientes, además de verificar expiración de token |
| **Decisión** | Implementar dos interceptors HTTP: `AuthInterceptor` (agrega token Basic o Bearer según contexto) y `TokenInterceptor` (verifica expiración y fuerza logout) |
| **Alternativas descartadas** | Interceptor único (mezcla responsabilidades); verificación solo en guards (requests directas bypassean) |
| **Justificación** | Separación de responsabilidades: uno para inyección de credenciales, otro para verificación de expiración; ambos registrados como `HTTP_INTERCEPTORS` en `CoreModule` |
| **Drivers** | AC-04, DA-06 |
| **Implementación** | `CoreModule.providers`: `AuthInterceptor` + `TokenInterceptor` como `HTTP_INTERCEPTORS` con `multi: true` |

### DA-13. Motor de Procesamiento Excel Reutilizable

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | La carga masiva de estudiantes y empleados comparte lógica de lectura de Excel pero difiere en el mapeo de filas |
| **Decisión** | Crear `ProcesadorExcelMotor` como motor genérico que acepta un `FilaExcelMapeador<T>` para convertir filas a objetos de comando |
| **Alternativas descartadas** | Procesamiento ad-hoc por entidad (duplicación); importación CSV (el cliente proporciona Excel); procesamiento asíncrono (complejidad adicional sin beneficio claro para el volumen esperado) |
| **Justificación** | Reutilización del código de lectura de workbook; extensible a nuevos tipos de carga sin modificar el motor; acoplamiento solo en la interfaz del mapeador |
| **Drivers** | AC-13, RF-AS-03, RT-14 |
| **Implementación** | `ProcesadorExcelMotor.procesarArchivo(MultipartFile, FilaExcelMapeador<T>)` → lista de comandos; mapeadores específicos: `ProcesadorArchivoEstudianteServicio`, `ProcesadorArchivoEmpleadoServicio` |

### DA-14. Manejo Centralizado de Excepciones

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Múltiples tipos de excepciones de dominio, seguridad e infraestructura deben traducirse a códigos HTTP apropiados |
| **Decisión** | `@ControllerAdvice` centralizado con mapa estático de tipo-de-excepción → código HTTP |
| **Alternativas descartadas** | Try-catch en cada controlador (duplicación); códigos HTTP en excepciones (acoplamiento dominio-HTTP) |
| **Justificación** | Único punto de traducción excepción→HTTP; las excepciones de dominio permanecen agnósticas de infraestructura; respuestas de error consistentes |
| **Drivers** | AC-07, AC-06 |
| **Implementación** | `ManejadorError extends ResponseEntityExceptionHandler` con `@ControllerAdvice`; mapa: `ValorInvalidoExcepcion → 400`, `AuthorizationException → 403`, `TecnicoExcepcion → 500`, etc. |

### DA-15. CORS Restringido a Origen Conocido

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | API REST consumida por SPA Angular; debe prevenir acceso desde orígenes no autorizados |
| **Decisión** | CORS configurado inline en `SecurityFilterChain` permitiendo solo el origen del frontend |
| **Alternativas descartadas** | CORS `*` (inseguro); CORS deshabilitado (no funciona con SPA en diferente puerto) |
| **Justificación** | Restringe acceso a API solo al frontend conocido; expone solo el header `Authorization` |
| **Drivers** | AC-06, RT-07 |
| **Implementación** | `CorsConfiguration.setAllowedOrigins(List.of("http://localhost:4200"))` con `AllowCredentials(true)` y `ExposedHeaders(["Authorization"])` |

### DA-16. API Stateless sin CSRF

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | API REST stateless con autenticación JWT no es vulnerable a CSRF porque no usa cookies de sesión |
| **Decisión** | Deshabilitar CSRF en Spring Security y SessionManagement stateless |
| **Alternativas descartadas** | CSRF habilitado con token sincronizado (innecesario para API JWT) |
| **Justificación** | CSRF es relevante solo para autenticación basada en cookies de sesión; JWT en header `Authorization` no es enviado automáticamente por el navegador |
| **Drivers** | AC-06 |
| **Implementación** | `http.csrf(AbstractHttpConfigurer::disable)` + `sessionManagement(SessionCreationPolicy.STATELESS)` |

### DA-17. Guards de Ruta con Validación de Rol en Frontend

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Aunque la autorización real ocurre en el backend, el frontend debe ocultar rutas y módulos no accesibles |
| **Decisión** | `SecurityGuard` que valida JWT y mapea rutas restringidas a roles permitidos; `PublicRouteGuard` que redirige usuarios autenticados fuera de páginas públicas |
| **Alternativas descartadas** | Sin guards (mala UX, errores 403 visibles); guard monolítico (difícil de mantener) |
| **Justificación** | UX limpia: el usuario no ve rutas que no puede acceder; defensa en profundidad (backend siempre valida, frontend solo filtra) |
| **Drivers** | AC-01, AC-05 |
| **Implementación** | `security.guard.ts` con `ROLE_ROUTES: Record<string, string[]>` mapeando paths a roles; `public-route.guard.ts` redirigiendo a `/home` si ya autenticado |

### DA-18. Visualización con Chart.js y Exportación con xlsx

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Los usuarios necesitan dashboards visuales y exportación de datos a Excel sin procesamiento del lado del servidor |
| **Decisión** | Chart.js 4.5.0 con plugin datalabels para visualización; librería xlsx 0.18.5 para generación client-side de archivos Excel |
| **Alternativas descartadas** | D3.js (curva de aprendizaje); Google Charts (dependencia externa); generación de Excel en backend (carga innecesaria en servidor) |
| **Justificación** | Chart.js es liviano y suficiente para gráficas de barras, tortas y líneas; xlsx permite generar Excel sin roundtrip al servidor; ambas librerías son de producción y bien mantenidas |
| **Drivers** | RF-AS-09, RF-AS-10, DN-02 |
| **Implementación** | `package.json`: `"chart.js": "^4.5.0"`, `"chartjs-plugin-datalabels": "^2.2.0"`, `"xlsx": "^0.18.5"` |

## 10. Glosario

| Término | Definición |
|---------|-----------|
| **Driver Arquitectónico** | Factor que influye significativamente en las decisiones de diseño del sistema (objetivo de negocio, atributo de calidad, restricción o riesgo) |
| **Atributo de Calidad** | Requisito no funcional medible que describe una propiedad del sistema (rendimiento, seguridad, mantenibilidad, etc.) |
| **Restricción Técnica** | Decisión no negociable impuesta por el contexto que limita las opciones de diseño |
| **Decisión Arquitectónica (ADR)** | Elección de diseño que responde a uno o más drivers y que tiene alternativas descartadas documentadas |
| **Escenario de Calidad** | Descripción estructurada de un atributo de calidad: fuente, estímulo, artefacto, entorno, respuesta y medida |
| **Riesgo Arquitectónico** | Punto donde las decisiones actuales podrían fallar si cambian los supuestos subyacentes |
| **Sensibilidad** | Propiedad del sistema especialmente afectada por cambios en una decisión arquitectónica particular |
| **CQRS** | Command Query Responsibility Segregation — separación estructural de operaciones de escritura y lectura |
| **Arquitectura Hexagonal** | Patrón de puertos y adaptadores que aísla el dominio de detalles de infraestructura y frameworks |
| **Motor de Validación** | Componente de dominio (`MotoresFabrica`) que encapsula reglas de validación diferenciadas por tipo de operación |
