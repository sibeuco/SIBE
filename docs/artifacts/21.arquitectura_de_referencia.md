# 21. Arquitectura de Referencia

## 1. PropГіsito y Alcance

Este documento describe la arquitectura de referencia del sistema SIBE (Sistema de InformaciГіn de Bienestar y EvangelizaciГіn), sirviendo como guГӯa tГ©cnica autoritativa para comprender, mantener y evolucionar la soluciГіn. EstГЎ construido a partir del anГЎlisis exhaustivo del cГіdigo fuente (backend y frontend) y de los artefactos de documentaciГіn del proyecto.

**Audiencia:** Arquitectos de software, desarrolladores, equipo de operaciones, auditores tГ©cnicos y stakeholders con interГ©s en la estructura tГ©cnica del sistema.

**Alcance:** Cubre las vistas arquitectГіnicas (contexto, contenedores, componentes, datos, seguridad, despliegue), las decisiones arquitectГіnicas implementadas, los atributos de calidad medibles, las restricciones tГ©cnicas y los riesgos arquitectГіnicos identificados.

## 2. Contexto del Sistema

### 2.1 DescripciГіn General

SIBE es una aplicaciГіn web diseГұada para digitalizar la gestiГіn de actividades y asistencia de la DirecciГіn de Bienestar y EvangelizaciГіn de la Universidad CatГіlica de Oriente (UCO). Reemplaza procesos manuales basados en planillas de papel, Google Forms, CheckPoint y AccuClass.

### 2.3 Stakeholders y Roles del Sistema

| Rol | Alcance | Permisos |
|-----|---------|----------|
| **Administrador de DirecciГіn** | Acceso completo a todas las ГЎreas y subГЎreas | CRUD completo + gestiГіn de usuarios + analГӯtica global |

| **Participante** | Actor externo (estudiantes, empleados, externos) | No accede al sistema; su asistencia es registrada por un operador |

| Backend вҶ’ SMTP | SMTP/STARTTLS | Spring Mail (Gmail puerto 587) |

### 5.2 CQRS вҖ” SeparaciГіn Comando/Consulta

SIBE implementa **CQRS estructural** en todas las capas. La separaciГіn no es eventual (sin event sourcing) sino paquetizada: las rutas de escritura y lectura son paquetes independientes en cada capa.

```mermaid
flowchart LR
    subgraph Escritura ["Escritura (Comando)"]
        direction TB
        CC[ComandoControlador] --> MC["ManejadorComandoRespuesta\n@Transactional"]
        MC --> UC["GuardarXUseCase /\nModificarXUseCase /\nEliminarXUseCase"]
        UC --> PC["RepositorioComando\n(puerto)"]
        PC --> IC[RepositorioComandoImpl]
    end

    subgraph Lectura ["Lectura (Consulta)"]
        direction TB
        CQ[ConsultaControlador] --> MQ["ManejadorRespuesta /\nManejadorParametroRespuesta\n@Transactional readOnly"]
        MQ --> UQ["ConsultarXUseCase /\nListarXUseCase"]
        UQ --> PQ["RepositorioConsulta\n(puerto)"]
        PQ --> IQ[RepositorioConsultaImpl]
    end

    IC --> DAO["DAO\n(JpaRepository)\ncompartido"]
    IQ --> DAO
```

**Flujo de un comando (escritura):**
1. `POST /api/actividades` вҶ’ `ActividadComandoControlador` (con `@PreAuthorize`)
2. вҶ’ `GuardarActividadManejador.ejecutar(ActividadComando)` (abre transacciГіn)
3. вҶ’ `GuardarActividadUseCase.ejecutar(Actividad)` (valida reglas, autorizaciГіn contextual)
4. вҶ’ `ActividadRepositorioComando.guardar(Actividad)` (puerto)
5. вҶ’ `ActividadRepositorioComandoImplementacion.guardar()` (mapea a entidad JPA, persiste)
6. вҶ’ Retorna `ComandoRespuesta<UUID>` con el identificador generado

**Flujo de una consulta (lectura):**
1. `GET /api/actividades/area/{id}` вҶ’ `ActividadConsultaControlador` (con `@PreAuthorize`)
2. вҶ’ `ConsultarActividadesPorAreaManejador.ejecutar(UUID)` (transacciГіn readOnly)
3. вҶ’ `ConsultarActividadesPorAreaUseCase.ejecutar(UUID)` (autorizaciГіn contextual)
4. вҶ’ `ActividadRepositorioConsulta.consultarPorArea(UUID)` (puerto)
5. вҶ’ `ActividadRepositorioConsultaImplementacion` (mapea entidades a DTOs)
6. вҶ’ Retorna `List<ActividadDTO>`

### 5.3 Cableado de Dependencias

Los use cases y servicios de dominio **no son Spring Beans**. Son POJOs instanciados explГӯcitamente mediante clases `@Configuration`:

- **`UseCaseBean.java`** вҖ” Declara cada use case como `@Bean` con inyecciГіn explГӯcita de puertos por constructor.
- **`ServiceBean.java`** вҖ” Declara cada servicio de dominio como `@Bean`.
- **`EncriptarClaveBean.java`** вҖ” Declara el `BCryptPasswordEncoder`.

Esto garantiza que el paquete `dominio` no tiene ninguna dependencia de Spring.

### 7.2 PatrГіn de Snapshot Temporal

SIBE implementa un **patrГіn de snapshot temporal** mediante la dualidad Miembro/Participante:

- **`Miembro`** representa los datos maestros actuales del individuo (actualizados semestralmente vГӯa carga masiva).
- **`Participante`** es un snapshot inmutable que captura el estado del miembro en el momento del registro de asistencia.
- **`RegistroAsistencia`** vincula un `Participante` con una `EjecuciГіnActividad` y es **inmutable** вҖ” nunca se actualiza ni elimina.

Esto garantiza que cambios posteriores en los datos maestros (ej. un estudiante cambia de programa acadГ©mico) no alteren el historial de asistencia.

### 10.1 EjecuciГіnActividad вҖ” Ciclo de Vida de EjecuciГіn


**Reglas:**
- FINALIZADA es **terminal e irreversible**. Los registros de asistencia son inmutables una vez creados.

**Efectos del estado INACTIVO:**

- No puede modificar perfil ni recuperar contraseГұa.

## 13. Decisiones ArquitectГіnicas (ADRs)

### DA-01 вҖ” Arquitectura Hexagonal con Puertos y Adaptadores

| Aspecto | Detalle |
|---------|---------|
| **DecisiГіn** | Backend en 3 capas: `dominio` (modelos, puertos, use cases, reglas) вҶ’ `aplicaciГіn` (handlers, fГЎbricas, comandos) вҶ’ `infraestructura` (controladores, DAOs, seguridad, config) |
| **Alternativas descartadas** | Arquitectura tradicional por capas (acoplamiento dominio-infra); Clean Architecture pura (ceremonia excesiva para el alcance) |
| **Medida de Г©xito** | 0 imports de Spring/JPA en paquetes `dominio` |
| **Drivers** | AC-07 Mantenibilidad, RT-11, DN-10 |

### DA-02 вҖ” CQRS Estructural en Todas las Capas

| Aspecto | Detalle |
|---------|---------|
| **DecisiГіn** | Separar controladores, handlers, puertos y use cases en paquetes `comando` y `consulta` en cada capa |
| **Alternativas descartadas** | Controlador Гәnico por recurso; CQRS con event sourcing (complejidad innecesaria) |
| **Drivers** | AC-08, RT-12 |

### DA-03 вҖ” Transacciones Declarativas en Interfaces de Handler

| Aspecto | Detalle |
|---------|---------|
| **DecisiГіn** | `@Transactional` en la interfaz `ManejadorComandoRespuesta<C,R>.ejecutar()`; `@Transactional(readOnly=true)` en interfaces de consulta |
| **Alternativas descartadas** | `@Transactional` en use cases (acoplarГӯa dominio a Spring); en repositorios (granularidad insuficiente) |
| **Drivers** | AC-12, AC-07, RT-13 |

### DA-04 вҖ” Motor de ValidaciГіn por OperaciГіn (MotoresFabrica)

| Aspecto | Detalle |
|---------|---------|
| **DecisiГіn** | Centralizar reglas de validaciГіn en motores estГЎticos pre-construidos en `MotoresFabrica`, ejecutados por `TipoOperacion` CREAR/MODIFICAR |
| **Alternativas descartadas** | Spring Validation en controladores; validaciГіn ad-hoc por use case |
| **Motores** | 13 motores (uno por entidad principal) con 28 implementaciones de `Regla` |
| **Drivers** | AC-07, AC-06 |

### DA-05 вҖ” GeneraciГіn de UUID con Loop Anti-ColisiГіn

| Aspecto | Detalle |
|---------|---------|
| **DecisiГіn** | `UtilUUID.generar(Predicate<UUID>)` verifica existencia en repositorio antes de asignar |
| **Drivers** | AC-07, AC-11 |

### DA-06 вҖ” Seguridad Stateless con JWT + HTTP Basic

| Aspecto | Detalle |
|---------|---------|
| **DecisiГіn** | AutenticaciГіn inicial vГӯa HTTP Basic; JWT firmado con HMAC-SHA256 conteniendo claims de identidad, rol y contexto organizacional |
| **Alternativas descartadas** | Sesiones servidor; OAuth2/OIDC (infraestructura excesiva) |
| **Claims** | email, id, authorities, rol, direccionId, areaId, subareaId |
| **ExpiraciГіn** | ~8.33 horas |
| **Drivers** | AC-04, RT-09, RT-10 |

### DA-07 вҖ” AutorizaciГіn de Doble Capa (Rol + Contexto Organizacional)

| Aspecto | Detalle |
|---------|---------|
