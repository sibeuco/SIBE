## 1. Gestión de Acceso y Cuentas

### HU-001 — Autenticación de Usuarios y Cierre de Sesión

| Campo | Detalle |
| ----- | ------- |
| **Código** | HU-001 |
| **Prioridad** | 1 |
| **Estado** | Completado |
| **Funcionalidad** | Gestión de Acceso y Cuentas |
| **Requisitos relacionados** | Inicio de sesión |

#### Descripción

**Como** usuario registrado (Administrador, Colaborador), **quiero** iniciar sesión de forma segura usando mi correo electrónico y contraseña, **para** poder acceder a las funcionalidades de SIBE que corresponden a mi rol.

#### Criterios de Aceptación

**Escenario 1: Inicio de sesión exitoso**

> **Dado que** soy un usuario registrado con credenciales válidas.
> **Cuando** ingreso mi correo electrónico y mi contraseña en el formulario de login y hago clic en "Iniciar Sesión".
> **Entonces** el sistema debe validar mis credenciales contra la base de datos, generar un Token de Acceso (JWT) y redirigirme al Dashboard principal de la aplicación.

**Escenario 2: Inicio de sesión fallido (Credenciales incorrectas)**

> **Dado que** soy un usuario registrado.
> **Cuando** ingreso mi correo electrónico y una contraseña incorrecta y hago clic en "Iniciar Sesión".
> **Entonces** el sistema debe rechazar la autenticación y debo permanecer en la página de login con un mensaje de error (ej. "Correo o contraseña inválidos").

**Escenario 3: Cierre de sesión**

> **Dado que** soy un usuario autenticado en el sistema.
> **Cuando** hago clic en el botón "Cerrar Sesión".
> **Entonces** el sistema debe invalidar mi sesión (eliminar el JWT del almacenamiento local) y redirigirme a la página de "Iniciar Sesión".

### HU-002 — Gestión de Cuentas de Usuario (Creación, Modificación y Deshabilitación)

| Campo | Detalle |
| ----- | ------- |
| **Código** | HU-002 |
| **Prioridad** | 2 |
| **Estado** | Completado |
| **Funcionalidad** | Gestión de Acceso y Cuentas |
| **Requisitos relacionados** | Creación de usuarios, Modificación de usuarios, Eliminación o deshabilitación de usuarios |

#### Descripción

**Como** Administrador de Dirección, **quiero** poder crear nuevas cuentas de usuario, modificar sus datos (incluyendo rol y área) y deshabilitar/eliminar cuentas, **para** gestionar el ciclo de vida de los accesos al sistema SIBE.

#### Criterios de Aceptación

**Escenario 1: Creación de un nuevo usuario (Flujo esperado)**

> **Dado que** soy un Administrador de Dirección en el formulario "Crear Usuario".
> **Cuando** ingreso los datos (nombres, apellidos, documento, correo, contraseña), selecciono un "Tipo de Usuario" (ej. "Colaborador") y selecciono el "Área" a la que pertenece de una lista desplegable poblada desde el backend, y hago clic en "Guardar".
> **Entonces** el sistema debe crear el nuevo usuario en la base de datos con el rol y el área asignados.

**Escenario 2: Modificación de un usuario**

> **Dado que** el usuario "Juan Pérez" existe y pertenece al "Área de Pastoral".
> **Cuando** lo edito y cambio su "Área" a "Área de Desarrollo Humano" y guardo los cambios.
> **Entonces** el sistema debe actualizar el registro de "Juan Pérez" con su nueva área.

**Escenario 3: Deshabilitación de usuario (Borrado Lógico)**

> **Dado que** el usuario "Juan Pérez" está asociado a actividades registradas (tiene historial).
> **Cuando** intento eliminar al usuario "Juan Pérez".
> **Entonces** el sistema no debe permitir un borrado físico, debe realizar un borrado lógico (ej. `user.setActivo(false)`) y "Juan Pérez" no debe poder iniciar sesión.

**Escenario 4: Eliminación de usuario (Borrado Físico)**

> **Dado que** el usuario "Usuario Temporal" existe y no tiene historial ni actividades asociadas.
> **Cuando** intento eliminar al usuario "Usuario Temporal".
> **Entonces** el sistema debe permitir un borrado físico.

**Escenario 5: Bloqueo por dependencia con HU-005**

> **Dado que** estoy en el formulario "Crear Usuario".
> **Cuando** intento seleccionar el "Área".
> **Entonces** la lista desplegable debe ser poblada por una llamada a la API (`GET /api/v1/areas`). Esta historia no puede moverse a "Terminada" hasta que la HU-005 (Consulta de Áreas) esté implementada en el backend y consumida por este formulario.

### HU-003 — Recuperación de Contraseña (Flujo de Olvido)

| Campo | Detalle |
| ----- | ------- |
| **Código** | HU-003 |
| **Prioridad** | 3 |
| **Estado** | Completado |
| **Funcionalidad** | Gestión de Acceso y Cuentas |
| **Requisitos relacionados** | Modificación de contraseña |

#### Descripción

**Como** usuario registrado (Administrador de Dirección, Administrador de Área o Colaborador), **quiero** iniciar un flujo de recuperación de contraseña cuando la he olvidado, **para** poder establecer una nueva contraseña segura y recuperar el acceso a mi cuenta SIBE.

#### Criterios de Aceptación

**Escenario 1: Solicitud de código de recuperación exitosa**

> **Dado que** soy un usuario que olvidó su contraseña y estoy en la página de "Iniciar Sesión".
> **Cuando** hago clic en el enlace "¿Olvidó su contraseña?" e ingreso mi correo electrónico que está registrado en SIBE.
> **Entonces** el sistema debe generar un código de 6 dígitos de recuperación único con tiempo de expiración de 5 minutos, enviar un correo electrónico con dicho código e instrucciones, y redirigirme a una vista para "Validar Código".

**Escenario 2: Solicitud de código con correo no registrado**

> **Dado que** estoy en la página de "Recuperar Contraseña".
> **Cuando** ingreso un correo electrónico que no está registrado en el sistema.
> **Entonces** el sistema debe mostrar un mensaje de notificación genérico (ej. "Si el correo está registrado, recibirás un código en breve"). Por seguridad, el sistema NO debe confirmar si el correo existe o no, y NO se debe enviar ningún correo electrónico.

**Escenario 3: Validación de código exitosa y cambio de contraseña**

> **Dado que** he recibido un código de recuperación válido en mi correo y estoy en la vista de "Validar Código".
> **Cuando** ingreso el código antes de que expire, soy redirigido a la vista de "Establecer Nueva Contraseña" e ingreso una nueva contraseña cumpliendo las políticas de seguridad.
> **Entonces** el sistema debe actualizar mi contraseña en la base de datos, invalidar el código para que no pueda ser reutilizado, y redirigirme a la página de "Iniciar Sesión" con un mensaje de éxito.

**Escenario 4: Validación de código fallida (inválido o expirado)**

> **Dado que** estoy en la vista de "Validar Código".
> **Cuando** ingreso un código que es incorrecto o ya ha expirado.
> **Entonces** el sistema debe mostrar un mensaje de error claro (ej. "El código es inválido o ha expirado. Por favor, solicita uno nuevo") y debo permanecer en la vista de "Validar Código".

**Escenario 5: Nueva contraseña no cumple políticas**

> **Dado que** he validado mi código y estoy en la vista de "Establecer Nueva Contraseña".
> **Cuando** ingreso una nueva contraseña que NO cumple las políticas de seguridad (ej. menos de 8 caracteres, sin números, etc.).
> **Entonces** el sistema debe mostrar un mensaje de error indicando los requisitos de la contraseña y debo permanecer en la vista.

#### Criterios de Aceptación

> **Dado que** estoy en la sección "Seguridad".

> **Entonces** el sistema debe mostrar un error de validación (ej. "Las nuevas contraseñas no coinciden") y la contraseña NO debe ser actualizada.

**Escenario 4: La nueva contraseña no cumple las políticas de seguridad**

> **Dado que** estoy en la sección "Seguridad".
> **Cuando** ingreso mi contraseña actual correctamente, ingreso una nueva contraseña débil (ej. "12345") y hago clic en "Actualizar Contraseña".
> **Entonces** el sistema debe mostrar un error de validación indicando los requisitos y la contraseña NO debe ser actualizada.

#### Descripción

**Como** Administrador de Dirección (al crear/editar un usuario), **quiero** poder consultar las listas de Direcciones, Áreas y Subáreas existentes en el sistema, **para** poder asignar correctamente un usuario a su Área o Subárea correspondiente en el formulario de gestión de usuarios.

#### Criterios de Aceptación

**Escenario 1: Poblar lista de Áreas en formulario de Creación de Usuario (Desbloqueo)**

> **Dado que** soy un Administrador de Dirección y estoy en el formulario de "Creación de Usuario".
> **Cuando** la vista del formulario se carga.
> **Entonces** el sistema debe hacer una llamada al backend (ej. `GET /api/areas`) para obtener la lista de todas las Áreas, y el campo desplegable "Área" debe poblarse con los nombres de las Áreas obtenidas (ej. "Pastoral", "Desarrollo Humano", etc.).

**Escenario 2: Asignar un Área a un nuevo usuario**

> **Dado que** el desplegable "Área" está poblado con datos del backend.
> **Cuando** selecciono un Área (ej. "Pastoral") del desplegable, completo el resto del formulario y hago clic en "Guardar".
> **Entonces** el sistema debe enviar el identificador (ID) del Área "Pastoral" junto con el resto de los datos del nuevo usuario al backend.

**Escenario 3: Consultar lista de todas las Direcciones**

> **Dado que** soy un usuario autenticado.
> **Cuando** solicito la lista de Direcciones (ej. `GET /api/direcciones`).
> **Entonces** el sistema debe devolver una lista de objetos `DireccionDTO`, cada uno conteniendo al menos su `id` y `nombre`.

**Escenario 4: Consultar lista de todas las Subáreas**

> **Dado que** soy un usuario autenticado.
> **Cuando** solicito la lista de Subáreas (ej. `GET /api/subareas`).
> **Entonces** el sistema debe devolver una lista de objetos `SubareaDTO`, cada uno conteniendo al menos su `id` y `nombre`.

**Escenario 5: Consultar identificador de un Área por su nombre**

> **Dado que** necesito el ID de un Área y solo conozco su nombre.
> **Cuando** consulto el sistema por el nombre "Desarrollo Humano" (ej. `GET /api/areas/search?nombre=Desarrollo Humano`).
> **Entonces** el sistema debe devolver el `id` correspondiente a esa Área.

| **Requisitos relacionados** | Gestión de acciones, Gestión de proyectos |

#### Descripción

**Como** Administrador de Dirección, **quiero** poder crear, modificar y consultar los Proyectos del Plan de Desarrollo, así como gestionar (crear, editar, consultar) las Acciones que componen dichos proyectos, **para** definir y estructurar el plan estratégico en el que se basarán los indicadores y las actividades.

#### Criterios de Aceptación

**Escenario 1: Creación de Acciones (Entidad Base)**

> **Dado que** soy un Administrador de Dirección en la sección "Configuración del Plan de Desarrollo".
> **Cuando** creo una nueva "Acción" proporcionando su Detalle (ej. "Realizar talleres de formación") y su Objetivo (ej. "Capacitar al personal").
> **Entonces** el sistema debe guardar esta nueva Acción y debe estar disponible para ser asignada a un Proyecto.

**Escenario 2: Creación de un Proyecto (Vinculando Acciones)**

> **Dado que** soy un Administrador de Dirección en la sección "Configuración del Plan de Desarrollo" y existen la "Acción A" y "Acción B".
> **Cuando** creo un nuevo "Proyecto" proporcionando Número ("P-01"), Nombre ("Proyecto Bienestar 2025") y Objetivo ("..."), selecciono "Acción A" y "Acción B" de la lista de acciones disponibles y guardo el Proyecto.
> **Entonces** el "Proyecto Bienestar 2025" debe ser creado y debe estar vinculado a "Acción A" y "Acción B".

**Escenario 3: Consulta de Proyectos**

> **Dado que** el "Proyecto Bienestar 2025" existe.
> **Cuando** consulto la lista de Proyectos.
> **Entonces** debo ver en la lista el Número ("P-01"), Nombre ("Proyecto Bienestar 2025") y su Objetivo.

**Escenario 4: Consulta de una Acción**

> **Dado que** la "Acción A" existe en el sistema.
> **Cuando** consulto el detalle de la "Acción A".
> **Entonces** el sistema debe mostrar su Detalle y su Objetivo.

**Escenario 5: Modificación de un Proyecto**

> **Dado que** estoy editando el "Proyecto Bienestar 2025" (que tiene "Acción A" y "Acción B").
> **Cuando** actualizo su Nombre a "Proyecto Bienestar 2026", desvinculo la "Acción B" y vinculo una nueva "Acción C", y guardo los cambios.
> **Entonces** el proyecto debe reflejar el nuevo nombre y sus acciones asociadas deben ser "Acción A" y "Acción C".

## 3. Administración de Participantes (Público Objetivo)

### HU-008 — Carga Masiva de Estudiantes desde Archivo Excel (.xlsx)

| Campo | Detalle |
| ----- | ------- |
| **Código** | HU-008 |
| **Prioridad** | 8 |
| **Estado** | Completado |
| **Funcionalidad** | Administración de Participantes (Público Objetivo) |
| **Requisitos relacionados** | Gestión de Participantes Internos |

#### Descripción

**Como** Administrador de Dirección, **quiero** cargar un archivo Excel (.xlsx) que contiene los datos de todos los estudiantes vigentes (pregrado y posgrado) proporcionado por el área de sistemas, **para** que la base de datos de SIBE esté actualizada semestralmente, asegurando que solo los participantes vigentes puedan registrar asistencia (vía RFID o documento).

#### Criterios de Aceptación

**Escenario 1: Carga Exitosa (Actualizar, Insertar y Desactivar)**

> **Dado que** soy un Administrador de Dirección en la sección "Gestión de Participantes" y tengo un archivo `estudiantes_2025-1.xlsx` con el formato y columnas correctas (ej. Documento, Nombres, Apellidos, Correo, Programa, Facultad, Codigo_RFID, etc.).
> **Cuando** selecciono la opción "Cargar Estudiantes" y subo dicho archivo.
> **Entonces** el sistema debe procesar el archivo en una única transacción, debe "Actualizar o Insertar" (Upsert) los datos de los estudiantes usando el Número de Documento como clave única, cualquier estudiante que exista en SIBE pero que NO venga en el nuevo archivo debe ser marcado como Inactivo (Borrado Lógico), y debo recibir un resumen de la operación (ej. "Carga exitosa: 2500 estudiantes actualizados, 300 nuevos creados, 150 marcados como inactivos").

**Escenario 2: Carga Fallida (Formato de Archivo Incorrecto)**

> **Dado que** soy un Administrador de Dirección.
> **Cuando** intento subir un archivo `estudiantes_malformato.xlsx` al que le falta una columna obligatoria (ej. "Documento").
> **Entonces** el sistema debe rechazar el archivo antes de intentar procesar los datos, debe mostrar un mensaje de error claro (ej. "Error al cargar el archivo") y la base de datos NO debe sufrir ninguna modificación.

**Escenario 3: Carga Fallida (Errores de Datos Internos)**

> **Dado que** soy un Administrador de Dirección.
> **Cuando** intento subir un archivo `estudiantes_datos_malos.xlsx` que contiene 1000 filas, pero 2 filas tienen el Documento duplicado dentro del mismo archivo, o un Correo en formato inválido.
> **Entonces** el sistema debe validar los datos y detectar las inconsistencias, la transacción completa debe revertirse (Rollback), debo recibir un reporte de errores (ej. "Carga fallida. Errores: Fila 50 (Documento duplicado), Fila 100 (Correo inválido)") y la base de datos debe quedar en el estado exacto en que estaba antes de la carga (Atomicidad).

### HU-012 — Registro de Asistencia en Vivo (Kiosco RFID/Documento)

| Campo | Detalle |
| ----- | ------- |

**Como** Colaborador (o responsable de la actividad), **quiero** iniciar una actividad programada y registrar la asistencia de los participantes a medida que llegan (vía RFID, Documento o como Externo), **para** tener un control de asistencia en tiempo real, seguro y eficiente, eliminando el papel.

#### Criterios de Aceptación

**Escenario 1: Iniciar el registro de una actividad**

> **Dado que** soy un Colaborador y he seleccionado la actividad "Taller de Liderazgo" (Ejecución ID: 201) que está `PENDIENTE`.
> **Cuando** hago clic en el botón "Iniciar Registro de Asistencia".

**Escenario 2: Registro Exitoso con RFID**

> **Dado que** la actividad "Taller de Liderazgo" (Ejecución ID: 201) está `EN_CURSO` y estoy en la interfaz de "Registro de Asistencia".
> **Cuando** un participante escanea su carnet y el lector RFID envía el código (RFID_VALIDO_123).
> **Entonces** el sistema debe llamar al endpoint de registro (ej. `POST .../ejecuciones/201/asistencia/rfid`), el backend debe buscar al Participante con ese RFID, crear un nuevo registro de Asistencia vinculando al Participante con la Ejecución 201, y la interfaz debe mostrar un mensaje de éxito (ej. "Participante agregado exitosamente") y limpiar el campo de entrada.

**Escenario 3: Registro Exitoso con Número de Documento**

> **Dado que** la actividad 201 está `EN_CURSO`.
> **Cuando** un participante no tiene carnet e ingreso su número de documento (1098765432) y presiono "Enter".
> **Entonces** el sistema debe llamar al endpoint de registro (ej. `POST .../ejecuciones/201/asistencia/documento`), el backend debe buscar al Participante con ese documento, crear el registro de Asistencia y la interfaz debe mostrar un mensaje de éxito (ej. "Participante agregado exitosamente").

**Escenario 4: Registro de Participante Externo**

> **Dado que** la actividad 201 está `EN_CURSO`.
> **Cuando** un asistente (ej. un acudiente) no existe en la base de datos SIBE, selecciono la opción "Registrar Externo" e ingreso Nombre Completo ("Maria Villegas") y Número de Documento ("987654321").
> **Entonces** el sistema debe llamar al endpoint de registro externo (ej. `POST .../ejecuciones/201/asistencia/externo`), el backend debe crear un registro de Asistencia guardando los campos `nombreExterno` y `documentoExterno`, y la interfaz debe mostrar un mensaje de éxito (ej. "Participante agregado exitosamente").

**Escenario 5: Participante no encontrado**

> **Dado que** la actividad 201 está `EN_CURSO`.
> **Cuando** ingreso un RFID (RFID_INVALIDO) o documento (000000) que no existe en la tabla Participante.
> **Entonces** el sistema debe mostrar un mensaje de error claro (ej. "Participante no encontrado. Verifique el dato o regístrelo como externo") y NO se debe crear ningún registro de asistencia.

**Escenario 6: Intento de Registro Duplicado**

> **Dado que** la actividad 201 está `EN_CURSO` y el participante "Carlos Rodríguez" (RFID_VALIDO_123) ya registró su asistencia.
> **Cuando** "Carlos Rodríguez" intenta escanear su carnet por segunda vez.
> **Entonces** el sistema debe detectar que ya existe un registro de asistencia para ese participante en esa ejecución, debe mostrar un mensaje de advertencia (ej. "Asistencia ya registrada para Carlos Rodríguez") y NO se debe crear un segundo registro.
