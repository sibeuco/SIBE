# 20. Alternativas de Solución

## 1. Propósito del Documento

Este artefacto presenta un análisis comparativo de las alternativas de mercado existentes para la gestión de actividades y asistencia en entornos universitarios, evaluadas frente a SIBE (Sistema de Información de Bienestar y Evangelización). El objetivo es documentar por qué las herramientas genéricas disponibles no satisfacen los requerimientos operacionales, institucionales y estratégicos de la Dirección de Bienestar y Evangelización de la Universidad Católica de Oriente (UCO), justificando así la necesidad de una solución a medida.

---

## 2. Contexto del Problema

La Dirección de Bienestar y Evangelización de la UCO gestiona una estructura organizacional jerárquica compleja:

```
Dirección de Bienestar y Evangelización
├── Área de Bienestar
│   ├── Deportes, Cancha Sintética, Gimnasio
│   ├── Extensión Cultural, Banda Sinfónica
│   └── Unidad de Salud, Acompañamiento Psicosocial, Trabajo Social
├── Área de Evangelización
├── Área de Hogar Santa María
└── Área de Servicio y Atención al Usuario
```

Esta Dirección necesita:
- **Digitalizar la gestión de asistencia** eliminando planillas físicas que generan pérdida de información, ilegibilidad y ralentización de procesos.
- **Planificar actividades semestralmente** con fechas programadas, indicadores estratégicos y colaboradores responsables.
- **Registrar asistencia de 3 tipos de participantes**: estudiantes internos, empleados internos y participantes externos.
- **Actualizar datos semestralmente** desde archivos Excel institucionales de matrícula y nómina.
- **Vincular actividades con el plan de desarrollo institucional** para rendición de cuentas y auditorías.
- **Generar reportes consolidados** con analítica segmentada por atributos académicos y laborales.
- **Controlar acceso diferenciado por área organizacional** según rol y contexto.

---

Los criterios utilizados para evaluar cada alternativa se derivan directamente de los requerimientos funcionales (RF), atributos de calidad (AC), drivers de negocio (DN) y requisitos de información (RI) documentados en los artefactos previos del proyecto:

| ID | Criterio | Descripción | Fuente |
|----|----------|-------------|--------|

## 4. Alternativas Evaluadas

### 4.1 Planillas Físicas en Papel (Situación Actual)

**Descripción:** Método manual actualmente utilizado por la Dirección. Los colaboradores registran la asistencia de los participantes en hojas impresas que se recopilan y archivan físicamente.

**Ventajas:**
- No requiere infraestructura tecnológica.
- Familiaridad del personal con el proceso.

**Limitaciones identificadas:**

| Pain Point | Impacto |
|------------|---------|
| Pérdida de información | Los registros físicos se extravían, deterioran o se vuelven ilegibles |
| Dificultades de lectura | Letra ilegible, deterioro físico del papel, manchas |
| Ralentización de procesos | El proceso manual consume tiempo excesivo del personal |
| Imposibilidad de análisis de datos | No permite consolidación, filtrado ni generación de indicadores |
| Dificultades para auditorías | Sin capacidad de generar reportes automáticos para rendición de cuentas |
| Impacto ambiental | Consumo de papel y costos operativos asociados |
| Sin trazabilidad estratégica | No vincula actividades con el plan de desarrollo institucional |

**Evaluación contra criterios:** No cumple ninguno (CE-01 a CE-12): **0/12**.

---

### 4.2 Google Forms

**Descripción:** Herramienta gratuita de Google para creación de formularios en línea. Actualmente utilizada por la Dirección como alternativa parcial para recopilar datos de asistencia.

**Ventajas:**
- Gratuita y accesible desde cualquier dispositivo.
- Permite recopilar respuestas en una hoja de cálculo (Google Sheets).
- Fácil de crear y compartir.

**Limitaciones frente a SIBE:**

| Criterio | Limitación |
|----------|-----------|
| CE-01 | No soporta lectores RFID ni hardware externo; el participante debe completar el formulario manualmente |
| CE-02 | No modela jerarquías organizacionales; todos los datos van a un repositorio plano sin distinción de área/subárea |
| CE-03 | No tiene control de acceso contextual; cualquier persona con el enlace puede ver o enviar respuestas. No permite restringir visibilidad por rol + área |
| CE-04 | No gestiona estados de actividad; no distingue entre una sesión planificada, en curso, finalizada o cancelada |
| CE-05 | No tiene mecanismo de carga masiva semestral; no sincroniza con bases de datos de matrícula/nómina universitaria |
| CE-06 | No vincula respuestas con perfiles académicos enriquecidos (programa, facultad, semestre, promedio, centro de costos) |
| CE-07 | Las respuestas pueden editarse o eliminarse, no garantiza inmutabilidad del historial |
| CE-08 | No tiene concepto de plan de desarrollo institucional ni indicadores estratégicos |
| CE-09 | No ofrece dashboards con filtros institucionales; la analítica requiere procesamiento manual externo |
| CE-10 | Cada formulario es independiente; no modela múltiples ejecuciones de una misma actividad |
| CE-11 | Soportaría participantes externos, pero sin distinguir tipo de participante |
| CE-12 | Exporta a CSV/Sheets pero sin estructura institucional (por Dirección, Área, Subárea) |

**Evaluación contra criterios: 0/12** (CE-11 parcial, no discrimina tipo).

### 4.3 CheckPoint

**Descripción:** Aplicación de control de asistencia diseñada para entornos educativos. Permite registrar la presencia de estudiantes en clases o eventos mediante dispositivos móviles.

**Ventajas:**
- Orientada específicamente a registro de asistencia.
- Permite registro por geolocalización o código QR.
- Interfaz diseñada para entornos educativos.

**Limitaciones frente a SIBE:**

| Criterio | Limitación |
|----------|-----------|
| CE-01 | No soporta lectura de carnet RFID universitario; usa métodos propios (geolocalización, QR) |
| CE-02 | No modela la jerarquía organizacional específica de una Dirección universitaria con áreas y subáreas |
| CE-03 | Su modelo de permisos está orientado a profesor/estudiante, no a Administrador de Dirección/Área/Colaborador con contexto organizacional |
| CE-04 | No gestiona el ciclo completo de actividades institucionales con estados de ejecución |
| CE-05 | No contempla la actualización semestral masiva de participantes desde archivos institucionales |
| CE-06 | No captura datos académicos y laborales enriquecidos del contexto universitario |
| CE-07 | No implementa snapshots inmutables del perfil del participante al momento del registro |
| CE-08 | No tiene vinculación con planes de desarrollo institucional |
| CE-09 | Analítica limitada a asistencia de clase; no ofrece filtros por programa académico, centro de costos o indicador estratégico |
| CE-10 | Diseñada para sesiones de clase, no para múltiples ejecuciones de actividades de bienestar |
| CE-11 | Orientada a estudiantes matriculados; registro de externos no es un caso de uso nativo |
| CE-12 | Exportaciones limitadas y sin estructura por jerarquía organizacional |

**Evaluación contra criterios: 0/12**.

### 4.4 AccuClass

**Descripción:** Software de gestión de asistencia para instituciones educativas. Proporciona funcionalidades de registro, seguimiento y generación de reportes de asistencia estudiantil.

**Ventajas:**
- Especializada en asistencia educativa.
- Generación de reportes de asistencia.
- Integración con algunos LMS.

**Limitaciones frente a SIBE:**

| Criterio | Limitación |
|----------|-----------|
| CE-01 | No integra lectura de carnet RFID universitario; utiliza métodos propios de identificación |
| CE-02 | Diseñada para estructura curso/profesor, no para jerarquía Dirección → Área → Subárea de bienestar |
| CE-03 | Roles limitados al contexto académico (profesor/administrador de curso), sin autorización contextual por área organizacional |
| CE-04 | Focalizada en asistencia a clases; no modela estados de actividades institucionales de bienestar |
| CE-05 | No soporta carga masiva semestral desde archivos Excel institucionales con upsert por identificación |
| CE-06 | Datos limitados al ámbito académico (curso, sección); no captura relación laboral, centro de costos ni indicadores estratégicos |
| CE-07 | No garantiza inmutabilidad de snapshots de participantes al momento del registro |
| CE-08 | Sin concepto de plan de desarrollo institucional ni trazabilidad actividad → indicador → proyecto |
| CE-09 | Reportes enfocados en asistencia académica; sin analítica consolidada con filtros institucionales |
| CE-10 | Modelo orientado a sesiones de clase programadas, no a múltiples ejecuciones de actividades de bienestar |
| CE-11 | Diseñada para estudiantes inscritos; no contempla participantes externos |
| CE-12 | Exportaciones con formato propio, sin estructura por Dirección/Área/Subárea |

**Evaluación contra criterios: 0/12**.

### 4.5 Trello

**Descripción:** Herramienta de gestión de proyectos basada en tableros Kanban. Permite organizar tareas mediante tarjetas, listas y etiquetas. Ampliamente utilizada para gestión de flujos de trabajo.

**Ventajas:**
- Interfaz visual intuitiva con tableros Kanban.
- Flexible y personalizable mediante etiquetas, listas y power-ups.
- Colaboración en tiempo real entre miembros del equipo.
- Plan gratuito disponible.

**Limitaciones frente a SIBE:**

| Criterio | Limitación |
|----------|-----------|
| CE-01 | No soporta hardware RFID ni registro de asistencia; es una herramienta de gestión de tareas, no de asistencia |
| CE-02 | Los tableros y espacios de trabajo no modelan jerarquías organizacionales universitarias; la estructura es plana |
| CE-03 | Los permisos son por tablero/espacio de trabajo, no por contexto organizacional (Dirección/Área/Subárea) |
| CE-04 | Las tarjetas tienen estados tipo Kanban (columnas), no un ciclo de vida de actividades con ejecuciones asociadas |
| CE-05 | No tiene mecanismo de carga masiva de participantes desde archivos institucionales |
| CE-06 | No modela perfiles de participantes con datos académicos/laborales; las tarjetas representan tareas, no personas |
| CE-07 | No tiene concepto de registro de asistencia con snapshot inmutable |
| CE-08 | No vincula tarjetas con indicadores de un plan de desarrollo institucional |
| CE-09 | Analítica limitada a métricas de productividad de tareas; no ofrece dashboards de asistencia con filtros institucionales |
| CE-10 | Las tarjetas representan tareas únicas; no modelan múltiples ejecuciones de una actividad recurrente con asistentes |
| CE-11 | No aplica; Trello no gestiona participantes/asistentes a eventos |
| CE-12 | Exporta datos de tableros, no reportes de asistencia estructurados por organización |

**Evaluación contra criterios: 0/12**.

### 4.6 Microsoft Power BI

**Descripción:** Plataforma de inteligencia de negocios de Microsoft para visualización de datos, creación de dashboards y generación de reportes analíticos.

**Ventajas:**
- Capacidades avanzadas de visualización y analítica.
- Conectores a múltiples fuentes de datos.
- Dashboards interactivos y compartibles.
- Integración con ecosistema Microsoft 365.

**Limitaciones frente a SIBE:**

| Criterio | Limitación |
|----------|-----------|
| CE-01 | No es un sistema transaccional; no registra asistencia ni interactúa con hardware RFID |
| CE-02 | Es una herramienta de visualización, no gestiona jerarquías organizacionales operacionales |
| CE-03 | El control de acceso es a nivel de informe/espacio de trabajo, no por contexto organizacional operacional |
| CE-04 | No gestiona el ciclo de vida de actividades; solo visualiza datos ya existentes en otra fuente |
| CE-05 | No administra la carga de participantes; necesita que la fuente de datos ya esté gestionada en otro sistema |
| CE-06 | Puede visualizar datos enriquecidos si existen, pero no los captura ni gestiona |
| CE-07 | No es un sistema de registro; no crea ni preserva snapshots de asistencia |
| CE-08 | Podría visualizar relaciones si la fuente los modela, pero no gestiona la vinculación actividad → indicador |
| CE-09 | Excelente para analítica **si los datos existen estructurados**; pero no genera los datos, solo los consume |
| CE-10 | No aplica; Power BI no gestiona actividades ni ejecuciones |
| CE-11 | No aplica; no registra participantes |
| CE-12 | Exporta visualizaciones, pero depende de que otra herramienta provea los datos estructurados |

**Evaluación contra criterios: 0/12** (CE-09 podría ser complementario si SIBE es la fuente de datos).

**Nota:** Power BI es una herramienta de consumo analítico, no de gestión operacional. Podría funcionar como complemento de SIBE para analítica avanzada, pero no como sustituto.

## 6. Análisis de Brechas Clave

### 6.1 Brecha Estructural: Gestión vs. Visualización vs. Organización

Las alternativas evaluadas se clasifican en tres categorías que revelan sus limitaciones fundamentales:

| Categoría | Herramientas | Limitación raíz |
|-----------|-------------|-----------------|
| **Recolección de datos** | Google Forms, planillas de papel | Capturan datos sin contexto, validación ni flujos de negocio |
| **Gestión de tareas** | Trello | Organiza trabajo del equipo, no gestiona asistencia ni actividades institucionales |
| **Visualización** | Power BI | Consume datos existentes; no genera, valida ni gestiona la operación |
| **Almacenamiento tabular** | Excel/Sheets | Contenedor genérico sin reglas de negocio, integridad referencial ni control de acceso contextual |
| **Asistencia educativa** | CheckPoint, AccuClass | Diseñadas para clases académicas, no para actividades de bienestar con jerarquía organizacional |

Ninguna alternativa cubre el espectro completo que SIBE integra: **captura (RFID) → gestión (ciclo de vida) → control (acceso contextual) → vinculación (plan estratégico) → análisis (dashboards institucionales) → exportación (reportes estructurados)**.

### 6.2 Brecha de Dominio: Bienestar Universitario ≠ Asistencia Académica

Las herramientas educativas existentes (CheckPoint, AccuClass) modelan la relación **profesor → estudiante → clase**, mientras que la Dirección de Bienestar opera bajo un modelo completamente diferente:

- **Organizador:** Colaborador del área de bienestar (no un profesor).
- **Actividad:** Evento institucional de bienestar (deportivo, cultural, salud), no una clase curricular.
- **Participante:** Puede ser estudiante, empleado o externo, con datos tanto académicos como laborales.
- **Contexto:** Vinculado a indicadores del plan de desarrollo institucional, no a calificaciones.
- **Visibilidad:** Restringida por jerarquía organizacional (Dirección/Área/Subárea), no por departamento académico.

### 6.3 Brecha de Integración: Hardware RFID

El diferenciador técnico más significativo es la integración con el **lector RFID del carnet universitario** en modo kiosco. Ninguna alternativa evaluada soporta esta capacidad. Este mecanismo permite:
- Registro de asistencia en **< 2 segundos** por participante.
- Eliminación de errores de digitación.
- Experiencia de usuario fluida en eventos masivos.
- Fallback por número de documento para participantes sin carnet.

## 8. Justificación de la Solución a Medida

### ¿Por qué construir SIBE y no adoptar una herramienta existente?

1. **No existe un producto en el mercado que cubra el dominio completo.** La gestión de actividades de bienestar universitario con RFID, jerarquía organizacional, vinculación a planes de desarrollo, y analítica institucional consolidada es un nicho específico que ninguna herramienta genérica ni educativa aborda.

2. **Las alternativas resuelven problemas parciales y desconectados.** Usar Google Forms para captura + Excel para almacenamiento + Power BI para visualización + Trello para organización crearía un ecosistema fragmentado sin integridad referencial, sin control de acceso unificado y sin flujos de negocio automatizados.

3. **La integración RFID requiere un sistema nativo.** El registro rápido mediante carnet universitario es un requerimiento diferenciador (DN-05) que no puede resolverse con complementos o integraciones de herramientas genéricas.

4. **La seguridad institucional exige control de acceso contextual.** La doble capa de autorización (rol RBAC + contexto organizacional) es un atributo de calidad arquitectónicamente significativo (AC-05) que las herramientas genéricas no implementan.

5. **La trazabilidad estratégica es un requisito de rendición de cuentas.** La UCO necesita demostrar mediante auditorías la contribución de las actividades de bienestar a los indicadores del plan de desarrollo institucional (DN-04). Ninguna alternativa modela esta relación.

6. **El costo total de propiedad favorece la solución a medida.** Combinar múltiples herramientas (Forms + Sheets + BI + gestión) implica licencias, formación, integración manual y riesgo de errores; un sistema unificado reduce la complejidad operacional.

## 9. Complementariedad Potencial

Aunque las alternativas no sustituyen a SIBE, algunas podrían complementarlo:

| Herramienta | Rol complementario potencial |
|-------------|------------------------------|
| **Power BI** | Consumo avanzado de datos analíticos exportados por SIBE para reportes ejecutivos de alto nivel |
| **Excel** | Fuente de datos para la carga masiva semestral de participantes (.xlsx); formato de exportación de reportes |
| **Google Forms** | Podría usarse para encuestas de satisfacción post-actividad, independientes del registro de asistencia |

---

## 10. Conclusión

Las alternativas evaluadas — planillas físicas, Google Forms, CheckPoint, AccuClass, Trello, Power BI y Excel — presentan una limitación estructural común: **no fueron diseñadas para soportar la complejidad operacional de una Dirección universitaria multiárea con jerarquía organizacional, hardware RFID, actualización semestral de participantes, vinculación estratégica con un plan de desarrollo institucional, ni analítica segmentada por atributos académicos y laborales.**

SIBE fue concebido específicamente para cubrir estas brechas, siendo sus diferenciadores clave:
- Registro rápido y seguro por **número de identificación o carnet RFID**.
- **Modelo de datos enriquecido** de participantes (académico + laboral + externo).
- **Jerarquía organizacional nativa** con control de acceso contextual.
- **Vinculación directa** con el plan de desarrollo institucional.
- **Analítica consolidada** con filtros específicos del dominio universitario.
- **Gestión completa del ciclo de vida** de actividades con múltiples ejecuciones.

La decisión de construir una solución a medida no es una preferencia tecnológica, sino una consecuencia directa de los requerimientos funcionales, atributos de calidad y drivers de negocio que definen el dominio operacional de la Dirección de Bienestar y Evangelización de la UCO.
