## 1. Gestión de Usuarios y Autenticación

| Código | Requisito de Información | Datos Asociados | Estado |
| ------ | ------------------------ | --------------- | ------ |
| RI-001 | El sistema debe almacenar la información de las personas vinculadas como usuarios, incluyendo datos de identidad personal y credenciales de acceso. | `Persona`: identificador (UUID), nombres (50 car.), apellidos (50 car.), correo electrónico (100 car.). `Usuario`: correo, clave (encriptada BCrypt), estado activo (booleano). | Aprobado |
| RI-002 | El sistema debe registrar el tipo y número de documento de identificación de cada persona, soportando múltiples tipos de documento. | `Identificación`: número de identificación (12 car.). `Tipo de Identificación`: sigla (6 car.), descripción (30 car.). | Aprobado |
| RI-003 | El sistema debe almacenar los tipos de usuario con sus permisos granulares (CRUD) para implementar el control de acceso basado en roles. | `Tipo de Usuario`: código (30 car.), nombre (30 car.), permisos: crear (booleano), modificar (booleano), eliminar (booleano), consultar (booleano). Roles definidos: ADMINISTRADOR_DIRECCION, ADMINISTRADOR_AREA, COLABORADOR. | Aprobado |
| RI-004 | El sistema debe registrar la vinculación de cada usuario con su estructura organizacional correspondiente (Dirección, Área o Subárea) para aplicar reglas de autorización contextual. | `Usuario-Organización`: identificador de usuario, identificador de dirección (opcional), identificador de área (opcional), identificador de subárea (opcional). | Aprobado |
| RI-005 | El sistema debe almacenar las peticiones de recuperación de contraseña, incluyendo el código de verificación cifrado y su marca temporal para validar vigencia (5 minutos). | `Petición de Recuperación de Clave`: código (cifrado), correo electrónico (100 car.), fecha y hora de creación (LocalDateTime). | Aprobado |

## 2. Estructura Organizacional

| RI-006 | El sistema debe almacenar la estructura jerárquica de la Dirección de Bienestar y Evangelización, compuesta por Direcciones como nivel superior. | `Dirección`: identificador (UUID), nombre (70 car.). Relaciones: una dirección contiene múltiples áreas y puede tener actividades asignadas directamente. | Aprobado |

## 3. Miembros de la Comunidad Universitaria

| Código | Requisito de Información | Datos Asociados | Estado |
| ------ | ------------------------ | --------------- | ------ |
| RI-008 | El sistema debe almacenar la información base de todos los miembros de la comunidad universitaria que participan en actividades, identificándolos de forma única. | `Miembro`: identificador (UUID), nombre completo (100 car.), número de identificación (12 car.). Clasificación: Interno (Estudiante, Empleado) o Externo. | Aprobado |
| RI-009 | El sistema debe registrar la información demográfica y académica de los estudiantes, obtenida mediante carga masiva semestral desde archivos Excel. | `Estudiante` (extiende Miembro Interno): fecha de nacimiento, nacionalidad (50 car.), estado civil (15 car.), correo personal (100 car.), correo institucional (100 car.), programa académico (100 car.), facultad (50 car.), año de ingreso, semestre actual (5 car.), créditos aprobados, promedio general, estado académico (10 car.), modalidad de estudio (50 car.), tiempo de llegada a la universidad (minutos), medio de transporte (30 car.). Datos adicionales heredados: ID carnet (20 car.), sexo (1 car.), ciudad de residencia. | Aprobado |
| RI-010 | El sistema debe registrar la información laboral de los empleados de la universidad, obtenida mediante carga masiva semestral desde archivos Excel. | `Empleado` (extiende Miembro Interno): relación laboral (código 4 car., descripción 20 car.), centro de costos (código 6 car., descripción 100 car.). Datos adicionales heredados: ID carnet (20 car.), sexo (1 car.), ciudad de residencia. | Aprobado |
| RI-011 | El sistema debe permitir el registro de participantes externos (no vinculados a la universidad) con información mínima para su identificación. | `Externo` (extiende Miembro): nombre completo, número de identificación. No requiere datos adicionales de la jerarquía interna. | Aprobado |
| RI-012 | El sistema debe almacenar el catálogo de ciudades de residencia para clasificar a los miembros internos de la comunidad universitaria. | `Ciudad de Residencia`: identificador (UUID), descripción (30 car.). | Aprobado |
| RI-013 | El sistema debe almacenar el catálogo de relaciones laborales para clasificar el tipo de vinculación de los empleados con la universidad. | `Relación Laboral`: identificador (UUID), código (4 car.), descripción (20 car.). Ejemplos: docentes tiempo completo, medio tiempo, ocasionales, cátedra, administrativos, operativos, directivos. | Aprobado |
| RI-014 | El sistema debe almacenar el catálogo de centros de costos para clasificar la dependencia administrativa de cada empleado. | `Centro de Costos`: identificador (UUID), código (6 car.), descripción (100 car.). | Aprobado |

## 4. Actividades y Ejecución

| Código | Requisito de Información | Datos Asociados | Estado |
| ------ | ------------------------ | --------------- | ------ |
| RI-015 | El sistema debe almacenar la información de las actividades programadas por cada nivel organizacional, incluyendo su vinculación con la planificación estratégica. | `Actividad`: identificador (UUID), nombre (200 car.), objetivo (500 car.), semestre (6 car., ej: "2025-1"), ruta de insumos (3000 car.), fecha de creación (LocalDate), colaborador responsable (UUID, referencia a usuario), creador (UUID, referencia a usuario), indicador asociado (referencia a Indicador). Vinculación organizacional: una actividad se asigna a exactamente una Dirección, Área o Subárea mediante tablas de unión. | Aprobado |
| RI-016 | El sistema debe registrar las ejecuciones (fechas programadas) de cada actividad, con control de estados que permita gestionar el ciclo de vida completo de cada sesión. | `Ejecución de Actividad`: identificador (UUID), fecha programada (LocalDate), fecha de realización (LocalDate, nullable), hora de inicio (LocalTime, nullable), hora de fin (LocalTime, nullable), estado de actividad (referencia a Estado). Pertenece a una Actividad. | Aprobado |
| RI-017 | El sistema debe almacenar el catálogo de estados posibles para las ejecuciones de actividades, soportando el flujo de vida de cada sesión programada. | `Estado de Actividad`: identificador (UUID), nombre (15 car.). Estados definidos: **Pendiente**, **En Curso**, **Finalizada**. Nota: no existe un estado "Cancelada" en la base de datos; la acción de cancelar una ejecución "En Curso" devuelve su estado a "Pendiente". | Aprobado |

## 5. Participantes y Registro de Asistencia

| Código | Requisito de Información | Datos Asociados | Estado |
| ------ | ------------------------ | --------------- | ------ |
| RI-018 | El sistema debe almacenar la información de los participantes que asisten a las actividades, capturando una instantánea (snapshot) de sus datos al momento de la asistencia para preservar la información histórica independiente de cambios futuros en la base de miembros. | `Participante`: identificador (UUID), referencia a miembro. `Participante Interno`: ciudad de residencia, ID carnet (20 car.), sexo (1 car.). `Participante Estudiante`: estado civil, programa académico, facultad, año de ingreso, semestre actual, créditos aprobados, promedio general, estado académico, modalidad de estudio, tiempo de llegada, medio de transporte. `Participante Empleado`: relación laboral, centro de costos. `Participante Externo`: sin datos adicionales (hereda nombre e identificación del miembro). | Aprobado |
| RI-019 | El sistema debe registrar cada evento de asistencia, vinculando de forma única un participante con una ejecución de actividad específica. | `Registro de Asistencia`: identificador (UUID), referencia a ejecución de actividad, referencia a participante. La combinación de ejecución + participante debe ser única para evitar registros duplicados. | Aprobado |
