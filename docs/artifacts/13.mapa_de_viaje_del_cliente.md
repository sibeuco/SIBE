# Mapa de Viaje del Cliente — SIBE

---

## 1. Descripción y Metodología

### 1.1 Propósito

El **Mapa de Viaje del Cliente** (Customer Journey Map) documenta la experiencia completa de cada tipo de usuario al interactuar con el sistema SIBE, desde el punto de contacto inicial (acceso a la URL) hasta la consecución de sus objetivos funcionales. Este artefacto describe:

- Las **fases secuenciales** que atraviesa cada usuario
- Las **pantallas, componentes y acciones** disponibles en cada fase
- Las **emociones, puntos de dolor y momentos de satisfacción** esperados
- Los **endpoints y servicios** consumidos en cada interacción
- Las **diferencias de experiencia** según el rol del usuario

### 1.2 Metodología de Construcción

El mapa se construyó a partir de la documentación de producto, la definición de roles y la arquitectura del sistema:

- **Frontend (Angular):** Rutas, guards de seguridad, componentes de pantalla y servicios HTTP
- **Backend (Spring Boot):** Controladores REST, autorización por rol y servicio de autorización contextual
- **Documentación existente:** Artefactos 10 (Story Map), 11 (Matriz de Roles), 12 (Comandos por Impacto), 15 (Requerimientos), 16 (Historias de Usuario)

### 1.3 Actores del Sistema

| Actor | Rol Técnico | Descripción | Contexto Organizacional |
| ----- | ----------- | ----------- | ----------------------- |
| **Ana** | `ADMINISTRADOR_DIRECCION` | Directora de Bienestar y Evangelización | Acceso total a todas las áreas, subáreas y operaciones |
| **Carlos** | `ADMINISTRADOR_AREA` | Coordinador del Área de Bienestar | Acceso a su área y las subáreas que la componen |
| **Laura** | `COLABORADOR` | Profesional de la Subárea de Deportes | Acceso a su área asignada y sus subáreas, o a su subárea si se asigna directamente. Ejecución de actividades asignadas |
| **Visitante** | No autenticado | Cualquier persona que accede a la URL del sistema | Sin acceso — solo ve login y recuperación de contraseña |

## 3. Viaje Compartido: Autenticación y Acceso

Este viaje es común a todos los actores del sistema antes de acceder a sus funcionalidades específicas.

### 3.1 Fase 0 — Acceso al Sistema

```
┌─────────┐     ┌──────────────┐     ┌──────────────┐     ┌─────────────┐
│ Abre URL │────►│ Guard evalúa │────►│ Login Page   │────►│ Ingresa     │
│ del SIBE │     │ token en     │     │ (formulario) │     │ credenciales│
└─────────┘     │ sessionStorage│     └──────────────┘     └──────┬──────┘
                └──────────────┘                                   │
                                                                   ▼
                                                          ┌──────────────┐
                                                          │ LoginService │
                                                          │ .validarLogin│
                                                          │ (Basic Auth) │
                                                          └──────┬──────┘
                                                                  │
                                              ┌───────────────────┼───────────────────┐
                                              ▼                                       ▼
                                    ┌──────────────┐                        ┌──────────────┐
                                    │ ✅ Token JWT  │                        │ ❌ Error 401  │
                                    │ en response   │                        │ "Correo o     │
                                    │ header →      │                        │ contraseña    │
                                    │ sessionStorage│                        │ incorrectos"  │
                                    └──────┬───────┘                        └──────────────┘
                                           │
                                           ▼
                                    ┌──────────────┐
                                    │ Decodifica JWT│
                                    │ → UserSession │
                                    │ {correo, rol, │
                                    │  areaId, etc} │
                                    │ → StateService│
                                    │ → /home       │
                                    └──────────────┘
```

| Paso | Pantalla | Componente Angular | Acción del Usuario | Servicio/Endpoint | Respuesta del Sistema |
| ---- | -------- | ------------------ | ------------------ | ----------------- | --------------------- |
| 0.1 | — | `AppComponent` | Abre la URL raíz `/` | — | Redirige a `/login` |
| 0.2 | Login | `LoginComponent` | Visualiza formulario con logo UCO | — | Campos: correo (email, required), contraseña (required) |
| 0.3 | Login | `LoginComponent` | Ingresa correo y contraseña, clic "Ingresar" | `LoginService.validarLogin()` → `GET /api/login` con Basic Auth | JWT en header `Authorization` almacenado en `sessionStorage` |
| 0.4 | — | `LoginComponent` | — (automático) | `StateService.updateState()` | Decodifica JWT: `{email, id, authorities, rol, direccionId, areaId, subareaId}` |
| 0.5 | Home | — | — (automático) | `router.navigate(['/home'])` | Navega al dashboard principal |

**Emociones y Puntos de Atención:**

| Momento | Emoción | Punto de Dolor | Momento de Satisfacción |
| ------- | ------- | -------------- | ----------------------- |
| Acceso URL | 😐 Neutral | URL no es intuitiva / memorable | Logo institucional genera confianza |
| Formulario login | 😊 Confianza | — | Diseño limpio con imagen UCO, campos claros |
| Credenciales incorrectas | 😟 Frustración | Mensaje genérico "Correo o contraseña incorrectos" | Enlace visible "¿Olvidaste tu contraseña?" |
| Login exitoso | 😊 Satisfacción | — | Redirección inmediata al dashboard |

### 3.2 Fase 0B — Recuperación de Contraseña (Flujo Alternativo)

El flujo de recuperación es un **wizard multi-paso** con indicador visual de progreso (steps 1-2-3).

| Paso | Pantalla | Acción del Usuario | Servicio/Endpoint | Respuesta del Sistema |
| ---- | -------- | ------------------ | ----------------- | --------------------- |
| B.1 | Paso 1 | Ingresa correo electrónico | `POST /api/usuarios/recuperacion/solicitar/{correo}` | "Código enviado al correo" → avanza a Paso 2 |
| B.2 | Paso 2 | Recibe código por email → lo ingresa | `POST /api/usuarios/recuperacion/validarCodigo` body: `{codigo, correo}` | "Código validado correctamente" → avanza a Paso 3 |
| B.3 | Paso 3 | Ingresa nueva contraseña + confirmación | `PUT /api/usuarios/recuperacion/recuperarClave` body: `{correo, clave}` | "Contraseña cambiada exitosamente" → redirige a `/login` |

**Emociones:**

| Momento | Emoción | Punto de Dolor | Momento de Satisfacción |
| ------- | ------- | -------------- | ----------------------- |
| Olvidó contraseña | 😰 Ansiedad | Debe recordar su correo exacto | Flujo visible y paso a paso |
| Espera del email | 😐 Impaciencia | El email puede tardar o caer en spam | Confirmación visual "Código enviado al correo" |
| Código expirado (5 min) | 😡 Frustración | Tiempo limitado no es visible claramente | Puede volver al paso anterior y solicitar nuevo código |
| Contraseña cambiada | 😊 Alivio | — | Redirección automática a login con mensaje de éxito |

**Secuencia Detallada del Registro de Asistencia:**

| Paso | Estado | Acción del Usuario | Endpoint | Respuesta del Sistema |
| ---- | ------ | ------------------ | -------- | --------------------- |
| 3.1 | PENDIENTE | Selecciona fecha programada en el date-selector | `sessionStorage.setItem('selectedActivityInfo', ...)` | Carga info de la ejecución seleccionada |
| 3.2 | PENDIENTE | Clic "Iniciar Actividad" | `PUT /api/actividades/iniciar/{ejecucionId}` | Estado → EN_CURSO, `horaInicio` capturada |
| 3.3 | EN_CURSO | Escanea carnet RFID → código aparece en campo | — | Campo "Código RFID" se llena automáticamente |
| 3.4 | EN_CURSO | Press Enter o clic "Buscar" | `GET /api/miembros/carnet/{rfid}` | Miembro encontrado → se agrega a la tabla |
| 3.5 | EN_CURSO | Ingresa documento manualmente | `GET /api/miembros/identificacion/{doc}` | Miembro encontrado → se agrega a la tabla |
| 3.6 | EN_CURSO | Participante no encontrado → confirma externo | Modal confirmación → modal datos externo | Crea `ParticipanteExterno` local en la lista |
| 3.7 | EN_CURSO | Clic "Finalizar Actividad" | `PUT /api/actividades/finalizar/{ejecucionId}` body: `[ParticipantRequest]` | Estado → FINALIZADA, participantes + asistencias creados en backend |
| 3.8 | EN_CURSO | Clic "Cancelar" → confirma en modal | `PUT /api/actividades/cancelar/{ejecucionId}` | Estado → PENDIENTE, lista de participantes se limpia |
| 3.9 | FINALIZADA | Ver lista de asistentes | `GET /api/actividades/ejecuciones/ejecucion/participantes/{ejecucionId}` | Tabla de participantes en modo solo lectura |

**Validaciones en el Frontend:**

| Validación | Momento | Comportamiento |
| ---------- | ------- | -------------- |
| Solo el colaborador asignado puede iniciar | Al cargar la pantalla | Si `actividad.colaborador ≠ token.id` → muestra mensaje de error y oculta botones de acción |
| Participante duplicado | Al buscar miembro | Si `documentoIdentificacion` ya está en la lista → muestra alerta "ya registrado" |
| Lista vacía al finalizar | Al clic "Finalizar" | Si `miembrosAsistencia.length === 0` → muestra alerta "No hay participantes" |
| Ejecución ya finalizada | Al clic en fecha programada | Si estado es FINALIZADO → modo solo lectura, sin campos de búsqueda ni botones de acción |
